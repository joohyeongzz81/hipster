# 가중치 재계산 시 쓰기 증폭 줄이기

## 문제

유저 가중치 재계산 배치는 처음에는 단순한 성능 개선 주제로 보였다.  
하지만 실제로 더 큰 문제는, 유저 가중치가 바뀔 때마다 과거 평점 row에 가중 반영값을 다시 써야 하는 구조였다.

이 구조를 유지하면 아래 비용이 함께 커진다.

- 유저 가중치 1회 변경
- 해당 유저가 남긴 과거 평점 row 다수 재기록
- 릴리즈 단위 집계 재반영
- 차트 재계산으로 연쇄 전파

즉, 느리게 바뀌는 상위 신호 하나가 대규모 write amplification으로 번지는 구조였다.  
핵심은 배치를 더 빠르게 만드는 것이 아니라, 가중치 계산 계층이 더 이상 과거 평점 row를 직접 만지지 않도록 책임 경계를 다시 그리는 것이었다.

## 해결 방향

가중치 재계산 배치의 책임을 아래처럼 축소했다.

- 가중치 계산 결과 저장
- 계산 근거 통계 저장
- downstream 집계 계층이 이후 배치 주기에서 최신 값을 흡수하도록 분리

즉, 가중치 배치는 결과와 근거만 관리하고, 평점 집계와 차트는 별도의 파생 집계 계층이 맡도록 구조를 나눴다.

이 과정에서 전수 로딩 구조도 함께 정리했다.

- Java 메모리 순회 기반 계산 -> SQL 집계 전환
- 전체 유저 한 번에 로딩 -> Spring Batch chunk 처리 전환
- 유저별 단건 처리 -> 일괄 쓰기 전환

## 구현

### 1. 과거 평점 row 재기록 구조 제거

가중치가 바뀌더라도 배치가 과거 평점 row를 직접 다시 쓰지 않도록 바꿨다.  
배치는 가중치 결과와 계산 근거 통계까지만 책임지고, 이후 릴리즈 단위 중간 집계가 최신 값을 흡수하도록 분리했다.

### 2. SQL 집계 기반 계산

유저별 통계를 애플리케이션 메모리에서 순회하지 않고 DB 집계로 계산했다.

- `COUNT(*)`
- `VAR_POP(score)`
- `AVG(review length)`
- `MAX(created_at)`

이렇게 바꾸면서 대용량 활동 기록을 가진 단일 유저에서도 메모리 폭증 없이 계산할 수 있게 됐다.

### 3. Spring Batch + chunk 처리

전체 유저를 한 번에 로드하던 구조를 chunk 기반 배치로 바꿨다.

- 실패 후 재시작 가능
- 영속성 컨텍스트 누적 완화
- 처리 단위 예측 가능

배치 내부에서는 청크 단위로 사용자 ID를 모아 통계를 읽고, 계산 결과와 근거 통계를 일괄 저장하도록 구성했다.

### 4. 계산 근거 통계 분리 저장

가중치 점수만 저장하면 운영자가 "왜 이 값이 나왔는가"를 설명하기 어렵다.  
그래서 계산 결과와 함께 근거 통계를 남겨, 이후 값 변화 설명과 산식 검증이 가능하도록 했다.

## 결과

- 단일 유저 평점 500,000건 기준 통계 계산: `10,420ms -> 1,138ms`
- 유저 50,000명 / 평점 5,000,000건 합성 데이터 기준 전체 배치 처리 시간: `921,000ms -> 359,200ms`
- 청크당 DB 통신 횟수: `2,000회 이상 -> 3~4회`

핵심 결과는 단순히 배치가 빨라진 것이 아니다.

- 쓰기 증폭을 유발하던 구조를 끊었다.
- 배치가 과거 평점 row를 직접 다시 쓰지 않게 만들었다.
- 가중치 계산을 설명 가능한 독립 계층으로 분리했다.
- 대량 유저 처리 시 메모리 폭증과 전수 로딩 리스크를 줄였다.

## 왜 이 구조가 중요했는가

이 프로젝트에서 가중치는 단순 보조 수치가 아니다.  
사용자의 평가가 이후 집계에서 얼마나 큰 영향력을 갖는지 결정하는 상위 신호다.

그래서 이 계층이 잘못 설계되면 문제가 바로 downstream으로 전파된다.

- 평점 집계 왜곡
- 차트 순위 왜곡
- 운영 설명 가능성 저하
- 느린 upstream 변경이 대규모 쓰기로 번지는 구조적 비용

결국 이 작업의 핵심은 "배치 최적화"보다 아래에 더 가깝다.

- 느리게 바뀌는 신호를 원본 row 재기록으로 풀지 않도록 경계를 재설계했다.
- 가중치 계산, 평점 집계, 차트 계층의 역할을 분리했다.

## 남은 한계

이 구조가 모든 문제를 닫은 것은 아니다.

- 가중치 변경이 평점 집계와 차트에 즉시 반영되지는 않는다.
- downstream 최신성은 이후 집계 배치 주기에 의존한다.
- 일부 correctness 이슈는 별도 후속 주제로 남아 있다.

즉, 이 문서의 포인트는 "완전히 닫힌 구조"라기보다,  
write amplification을 끊고 운영 가능한 책임 경계를 다시 그었다는 데 있다.
