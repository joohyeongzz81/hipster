# 유저 가중치 재계산 배치에서 쓰기 증폭 줄이기

> 유저 가중치 변경이 과거 평점 row 재기록으로 번지며 대규모 write amplification을 만들던 구조를 분리했습니다.  
> 가중치 재계산 배치는 `users.weighting_score`, `user_weight_stats`만 관리하도록 축소했고, 이후 평점 집계 배치가 최신 가중치를 읽어 `release_rating_summary`를 다시 계산하도록 책임을 재배치했습니다.

이 문서에서 보여드리고 싶은 핵심은 단순한 배치 성능 개선이 아닙니다.  
느리게 바뀌는 유저 가중치 계산을 과거 평점 row 재기록과 분리하고, 설명 가능한 계산 근거와 운영 가능한 배치 구조로 책임을 다시 나눈 과정입니다.

---

## 1. 문제 상황

초기 구조에서는 유저 가중치가 바뀌면 그 유저가 과거에 남긴 평점 row마다 가중 반영값도 다시 써야 한다는 전제가 깔려 있었습니다.  
이 구조의 중심에는 `ratings.weighted_score` 같은 비정규화 컬럼이 있었습니다.

이 전제를 유지하는 한, 유저 가중치 변경은 평점 행 단위 재기록과 이후 집계·차트 계산 비용 악화로 전파될 수밖에 없었습니다.

- 유저 가중치 1회 변경
- 해당 유저의 과거 rating row 다수 재기록
- 대규모 UPDATE에 따른 쓰기 비용과 락 부담 증가
- 이후 집계와 차트 계산 비용까지 함께 악화

즉, 느리게 바뀌는 upstream 신호 하나가 수백만 건의 write amplification으로 번지는 구조였습니다.

이 문제는 단순히 배치 SQL을 조금 더 튜닝한다고 해결되지 않았습니다.  
결국 과거 평점 row를 다시 쓰는 전제 자체를 버려야 했습니다.

---

## 2. 구조적 판단

제가 내린 핵심 판단은 두 가지였습니다.

첫째, 가중치 재계산 배치는 더 이상 과거 평점 row를 직접 갱신하지 않아야 했습니다.  
둘째, downstream 반영은 weighting 배치가 직접 처리하지 않고, 이후 평점 집계 배치가 최신 `users.weighting_score`를 읽어 `release_rating_summary`를 다시 계산하도록 분리해야 했습니다.

구조는 아래처럼 다시 나눴습니다.

- weighting 계층은 `users.weighting_score`, `user_weight_stats`만 관리합니다.
- rating 계층은 `ratings + users.weighting_score`를 읽어 `release_rating_summary`를 만듭니다.
- chart 계층은 `release_rating_summary`를 읽어 최종 차트를 만듭니다.

이렇게 바꾸면 유저 가중치 변화는 더 이상 per-rating row 재기록으로 반영되지 않고, 중간 집계 계층이 주기적으로 흡수하는 구조가 됩니다.

유저 가중치는 실시간 금융 정산처럼 즉시 반영이 필수인 값이 아니라, 일정 주기 안에서 수렴해도 서비스 경험에 큰 문제가 없는 정책 값으로 판단했습니다.  
그래서 weighting 배치가 downstream 집계 테이블까지 직접 재기록하는 대신, 이후 평점 집계 배치가 최신 가중치를 읽어 반영하도록 책임을 분리했습니다.

---

## 3. 해결 과정

### 3-1. 메모리 전수 로딩을 SQL 집계로 옮겼습니다

초기에는 유저 1명의 Rating, Review를 애플리케이션 메모리로 모두 끌고 와 평균, 분산, 리뷰 길이를 계산했습니다.  
구현은 단순했지만, 대용량 유저에서는 메모리 비용이 너무 컸습니다.

그래서 아래 계산을 SQL 집계로 이동했습니다.

- `COUNT(*)`
- `VAR_POP(score)`
- `AVG(review length)`
- `MAX(created_at)`

집계 방식을 바꾸기 전에, 단일 유저 평점 500,000건 더미 데이터 기준으로 Java Stream 집계와 SQL 집계가 같은 통계 값을 내는지 먼저 비교했습니다.
평점 수, 분산, 평균 리뷰 길이, 마지막 활동일은 부동소수점 허용 오차 `epsilon = 1e-6` 범위 내에서 동등함을 확인한 뒤, 성능과 메모리 사용량을 측정했습니다.

이 변경으로 유저 가중치 산출을 원천 데이터 전수 로딩 없이 계산할 수 있게 되었고, 계산 근거를 통계 집합으로 정리할 수 있게 되었습니다.

### 3-2. 전수 로딩 스크립트를 청크 배치로 바꿨습니다

유저 전체를 한 번에 로드하던 구조는 Heap 상한을 쉽게 건드렸고, 실패 시 중간부터 다시 시작하기도 어려웠습니다.

그래서 Spring Batch 기반 청크 처리로 전환했습니다.

- 성공 청크를 보존할 수 있게 했습니다.
- 실패 지점 이후 다시 시작할 수 있게 했습니다.
- 영속성 컨텍스트 누적을 줄였습니다.

즉, 가중치 재계산을 한 번에 끝내는 스크립트가 아니라 중단 가능하고 재시작 가능한 배치 잡으로 바꿨습니다.

### 3-3. 유저별 단건 처리를 청크 단위 일괄 처리로 바꿨습니다

청크 처리로 바꿔도, 청크 내부에서 유저별 단건 SELECT를 반복하면 DB 왕복이 다시 병목이 됩니다.

이 배치에서는 Processor가 유저별 통계를 개별 처리하기보다, Writer가 청크 전체 `userIds`를 기준으로 집계를 한 번에 읽고 결과를 일괄 반영하도록 설계했습니다.  
이를 통해 유저별 반복 SELECT를 줄이고, 청크 단위의 읽기/쓰기 비용을 예측 가능한 수준으로 고정할 수 있었습니다.

그래서 현재 구조에서는 Writer가 청크 전체를 한 번에 처리합니다.

1. 청크에서 `userIds`를 수집합니다.
2. `ratings`, `reviews` 집계를 `IN (:userIds)` SQL로 읽습니다.
3. 메모리에서 가중치를 계산합니다.
4. `user_weight_stats`를 batch upsert 합니다.
5. `users.weighting_score`를 일괄 갱신합니다.

이 단계에서 중요한 점은 결과값만 저장하지 않았다는 것입니다.  
`user_weight_stats`를 함께 저장해 계산 근거를 남기고, 다음 배치 품질 비교와 이상치 분석이 가능하도록 만들었습니다.

### 3-4. 45일 주기 판정을 배치 메타데이터 기준으로 관리했습니다

스케줄러는 매일 새벽 2시에 돌지만, 실제 Job 실행은 45일이 지났을 때만 수행합니다.

이때 아래 두 가지를 함께 사용했습니다.

- `JobExplorer`로 마지막 성공 실행일 확인
- `ShedLock`으로 멀티 인스턴스 중복 실행 방지

즉, 45일마다 실행을 크론 표현식에만 의존하지 않고, 실제 배치 실행 이력을 기준으로 주기를 판정하도록 구성했습니다.

유저 활동성과 평점 성향은 일 단위로 급격히 변하는 값이 아니라고 보고, 재계산 비용과 수렴 주기를 함께 고려해 45일 주기로 운영했습니다.  
이 주기는 [RYM Weighting FAQ](https://rateyourmusic.com/wiki/RYM:Weighting%20FAQ)를 참고했으며, 이를 그대로 복제하기보다 현재 서비스 규모와 운영 비용에서 무리가 없다고 판단해 채택했습니다.

## 4. 핵심 성과

| 관점 | 측정 조건 | 개선 전 | 개선 후 |
| --- | --- | --- | --- |
| 유저 통계 계산 시간 | 단일 유저 평점 500,000건 | 10,420ms | **1,138ms** |
| 유저 통계 계산 시 Heap 변동 | 단일 유저 평점 500,000건 | 512MB 이상 폭증 | **10MB 미만** |
| 전체 배치 처리 시간 | 유저 50,000명 / 평점 5,000,000건 합성 데이터 | 921,000ms | **359,200ms** |
| 전체 배치 Heap 피크 | 유저 50,000명 / 평점 5,000,000건 합성 데이터 | 498MB | **386MB** |
| 청크당 DB 통신 횟수 | 1,000명 청크 기준 | 2,000회 이상 | **3~4회** |

아래 수치는 로컬 포트폴리오 환경의 Spring Boot + Docker MySQL 8.0 기준 합성 데이터셋으로 측정했습니다.  
실행 시간은 AOP 기반으로 측정했고, DB 통신 횟수는 Hibernate SQL 로그 기준으로 확인했으며, Heap 피크는 실행 중 JVM 메모리 사용량 기준으로 관측했습니다.

이 수치는 단순한 배치 속도 개선이 아니라, 느리게 바뀌는 정책 값을 과거 평점 row 재기록에 연결하던 구조적 전제를 제거한 결과를 보여줍니다.

- 유저 가중치 변경을 과거 평점 row 재기록과 분리해 잘못된 전제를 제거했습니다.
- 유저 가중치 계산을 메모리 중심 처리에서 DB 집계 중심 처리로 옮겼습니다.
- 전수 로딩 구조를 청크 배치 구조로 바꿨습니다.
- 유저별 단건 처리 대신 일괄 쓰기 구조로 DB 왕복 비용을 줄였습니다.
- 성능 개선은 그 결과로 따라왔고, Heap 사용량과 재시작 가능성도 함께 개선했습니다.

---

## 6. 최종적으로 얻은 것

- 유저 가중치 변경을 과거 평점 row 재기록과 분리해 write amplification의 구조적 원인을 제거했습니다.
- 45일 배치가 downstream 대량 쓰기를 직접 일으키지 않게 만들었습니다.
- 유저 가중치 계산을 설명 가능한 통계 기반 프로세스로 바꿨습니다.
- 대량 유저를 처리해도 메모리 폭증과 전수 로딩 리스크를 줄였습니다.
- 실패 후 재시작, 멀티 인스턴스 중복 실행 방지 같은 운영 안정성을 확보했습니다.
- 유저 가중치 계층, 평점 집계 계층, 차트 계층의 책임 경계를 더 명확히 분리했습니다.

---

## 7. 남겨둔 것

- 가중치 변경이 `release_rating_summary`와 `chart_scores`에 즉시 반영되지는 않습니다.
- Anti-Entropy와 차트 배치 주기를 함께 이해해야 전체 데이터 흐름이 보입니다.
- 45일 주기는 [RYM Weighting FAQ](https://rateyourmusic.com/wiki/RYM:Weighting%20FAQ)를 참고해 채택했으며, 정책 비교와 세부 근거는 별도 문서에서 더 보강할 수 있습니다.

하지만 이 비용은 의식적으로 받아들인 것입니다.  
하나의 배치가 모든 파생 집계 테이블을 직접 다시 쓰게 만드는 구조보다, 책임 분리와 주기적 수렴이 현재 규모와 운영 복잡도에 더 맞는 선택이었습니다.
