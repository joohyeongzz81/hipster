# 유저 가중치 변경이 만드는 쓰기 증폭을 줄이기 위한 구조 재설계

처음에는 유저 가중치 배치의 성능 병목을 줄이는 일이 목표였습니다. 하지만 실제 병목은 가중치 변경이 과거 원본 평점 재기록으로 번지는 구조에 있었습니다. 이 문서는 배치 최적화 문제로 보였던 것을 직접 쓰기 범위와 책임 경계의 문제로 다시 정의하고, 가중치 배치의 역할을 좁히는 방향으로 구조를 바꾼 과정을 정리합니다.

---

## 1. 문제 상황

처음 문제의식은 유저 가중치 배치가 느리다는 것이었습니다. 하지만 병목을 줄이고 나서도 가중치가 한 번 바뀔 때마다 과거 평점들을 다시 써야 하는 전제는 남아 있었습니다.

초기 구조에서는 유저 가중치가 바뀌면 그 유저가 남긴 원본 평점 행마다 가중 반영값도 다시 써야 했습니다. `ratings.weighted_score` 같은 비정규화 컬럼을 두고 있었기 때문입니다. 이 전제를 유지하면 가중치 한 번의 변경이 아래 비용으로 이어졌습니다.

- 해당 유저의 원본 평점 행 다수 재기록
- 대규모 UPDATE에 따른 쓰기 비용과 락 부담 증가
- 가중치 변경 1회가 다수 행 UPDATE로 번져 배치 처리 범위가 계속 커짐

특히 이런 배치가 `ratings` 행 락을 오래 잡기 시작하면, 같은 사용자의 평점 수정이나 삭제 같은 실제 쓰기 요청도 저장 지연이나 락 대기로 이어질 수 있었습니다.

문제는 배치 하나가 느린 데서 끝나지 않았습니다. 천천히 바뀌는 값 하나가 대량 UPDATE를 반복하게 만드는 한, 전수 로딩을 줄이거나 배치 스레드 수를 조정해도 본질적인 쓰기 부담은 남았습니다. 결국 원본 평점 행을 다시 쓰는 전제 자체를 끊어야 했습니다.

---

## 2. 판단 기준

중요했던 건 가중치를 얼마나 자주 다시 계산하느냐보다, 가중치 변경이 어느 경계까지 직접 쓰기를 유발하느냐였습니다. `ratings.weighted_score`를 유지한 채 배치만 더 빠르게 만드는 것으로는 원본 평점 행 재기록 문제를 없앨 수 없었습니다. 그래서 가중치 배치가 원본 평점 계층과 평점 집계 계층까지 한 번에 책임지지 않도록, 직접 쓰기 범위와 후속 반영 경계를 다시 나눴습니다.

기준은 세 가지였습니다.

- 가중치 재계산 배치는 원본 평점 행을 직접 갱신하지 않습니다.
- 가중치 변경 이후 평점 집계 계층 반영은 가중치 배치가 끝까지 끌고 가지 않습니다.
- 평점 집계 경로가 최신 `users.weighting_score`를 읽어 `release_rating_summary`를 다시 계산합니다.

### 2-1. 최종 구조

| 계층 | 책임 |
| --- | --- |
| 가중치 계층 (`users.weighting_score`, `user_weight_stats`) | 유저별 최신 가중치와 계산 근거 통계 보존 |
| 원본 평점 계층 (`ratings`) | 평점 원본과 변경 이력 보존 |
| 평점 집계 계층 (`release_rating_summary`) | 릴리즈 단위 집계 결과 보존 및 최신 유저 가중치 기준 재계산 |
| 차트 계층 (`chart_scores`) | 평점 집계 결과를 바탕으로 최종 차트 결과 생성 |

---

## 3. 구현 과정

작업 순서는 처음부터 구조 재설계가 아니었습니다. 배치가 느리다는 문제의식에서 시작해 성능 병목을 먼저 걷어냈고, 그 과정에서 어떤 쓰기 경로를 끊어야 할지가 핵심 질문이 됐습니다.

### 3-1. 배치 병목을 먼저 줄였지만, 원본 평점 재기록 문제는 남았습니다

초기 구현은 유저 1명의 평점과 리뷰를 애플리케이션 메모리로 모두 끌고 와 평균, 분산, 리뷰 길이를 계산했습니다. 데이터가 많은 유저에서는 메모리 비용이 너무 컸기 때문에, 이를 `COUNT(*)`, `VAR_POP(score)`, `AVG(review length)`, `MAX(created_at)` SQL 집계로 옮겼습니다. 단일 유저 평점 500,000건 기준으로 계산 시간은 10,420ms → 1,138ms, 힙 변동은 512MB 이상 → 10MB 미만으로 줄었습니다.

또한 유저 전체를 한 번에 로드하던 스크립트를 `Spring Batch` 기반 청크 배치로 바꿔, 실패 지점 이후 재시작이 가능하도록 만들었습니다. 여기에 `Writer`가 청크 전체 `userIds`를 기준으로 집계를 한 번에 읽고 결과를 일괄 반영하는 구조를 적용해, 청크당 DB 통신 횟수도 1,000명 기준 2,000회 이상 → 3~4회로 줄였습니다.

이 단계에서 줄어든 건 계산 비용과 배치 실행 비용이었습니다. 하지만 가중치 변경이 원본 평점 행 재기록으로 번지는 구조 자체는 그대로 남아 있었습니다.

### 3-2. 가중치 배치의 직접 쓰기 범위를 `users`, `user_weight_stats`로 제한했습니다

성능 병목을 줄인 뒤에도 가장 크게 남은 질문은 "왜 이 배치가 원본 평점 계층과 평점 집계 계층까지 책임져야 하지?"였습니다. 그 질문 끝에 `ratings.weighted_score` 같은 평점 행 단위 반영값을 유지하지 않기로 했습니다. 가중치 배치는 사용자별 통계와 최신 유저 가중치만 저장하고, 평점 집계 계층 반영은 평점 집계 경로가 최신 가중치를 읽어 `release_rating_summary`를 다시 계산하게 했습니다.

`user_weight_stats`를 함께 저장한 이유도 여기서 중요했습니다. 최신 유저 가중치만 남기면 배치 결과를 다음 실행과 비교하거나 이상치를 추적하기 어렵기 때문에, 가중치 계산 근거를 별도 통계로 남겨 직접 쓰기 범위를 좁히면서도 운영 가시성은 유지했습니다.

---

## 4. 결과

### 측정 조건

- 유저 통계 계산: 단일 유저 평점 500,000건 기준
- 전체 배치 처리: 유저 50,000명 / 평점 5,000,000건 합성 데이터 기준
- 모든 수치는 로컬 포트폴리오 환경 기준이며, 운영 RPS나 분산 환경을 대표하지 않습니다.

### 수치 요약

| 항목 | 변경 전 | 변경 후 |
| --- | --- | --- |
| 유저 통계 계산 시간 | 10,420ms | 1,138ms |
| 유저 통계 계산 시 힙 변동 | 512MB 이상 | 10MB 미만 |
| 전체 배치 처리 시간 | 921,000ms | 359,200ms |
| 전체 배치 힙 피크 | 498MB | 386MB |
| 청크당 DB 통신 횟수 (1,000명 기준) | 2,000회 이상 | 3~4회 |

### 구조적으로 달라진 것

| 비교 항목 | 이전 | 이후 |
| --- | --- | --- |
| 유저 가중치 1회 변경 시 직접 쓰기 범위 | `users` + 과거 `ratings.weighted_score` 다수 행 | `users.weighting_score`, `user_weight_stats` |
| 가중치 변경 이후 집계 반영 방식 | 가중치 변경 시 과거 원본 평점 행 재기록 필요 | 평점 집계 경로가 최신 가중치를 읽어 `release_rating_summary` 재계산 |
| 원본 평점 행 재기록 여부 | 필요 | 없음 |
| 장애 시 복구 범위 | 하나의 배치가 여러 파생 쓰기 경로와 얽힘 | 가중치 / 평점 집계 계층 / 차트 계층 (`chart_scores`)별로 복구 경로 분리 |
