# 평점 집계 계층을 분리하고 결과적 일관성으로 수렴시키기

처음에는 릴리즈 평균 평점 조회를 빠르게 만드는 일이 목표였습니다. 그런데 문제를 파고들수록, `ratings`에 조회 책임과 가중치 반영 책임이 함께 몰려 있다는 점이 더 큰 병목으로 드러났습니다. 그래서 `release_rating_summary`를 릴리즈 단위 평점 집계 계층으로 세우고, 집계 갱신은 비동기 반영과 전체 재집계 보정으로 결과적 일관성에 수렴시키는 구조로 바꿨습니다.

---

## 1. 문제 상황

초기 구조에서는 릴리즈 상세와 목록에서 평균 평점과 평점 수를 보여주기 위해 원본 `ratings`를 직접 읽고 집계했습니다. 평균 평점은 단순 부가 정보가 아니라 사용자가 어떤 앨범을 더 볼지 판단하는 공개 지표에 가깝습니다. 이 방식은 평점이 쌓일수록 조회 비용이 선형으로 증가했고, 조회 핫패스가 원본 테이블 스캔에 계속 묶여 있었습니다.

같은 계층에 가중치 반영 책임까지 얹으려 하면 쓰기 쪽도 무거워졌습니다. `ratings.weighted_score` 같은 비정규화 컬럼 전제를 유지하면, 가중치 변경 한 번이 과거 원본 평점 행 다수 재기록으로 번질 수밖에 없었습니다.

이 전제에서는 유저 가중치 변경이 아래 비용으로 전파됐습니다.

- 가중치 1회 변경 → 해당 유저의 과거 원본 평점 행 다수 재기록
- 대규모 UPDATE에 따른 쓰기 비용과 락 부담 증가
- 같은 정책 변경이 원본 데이터를 직접 다시 쓰는 범위를 계속 넓힘

조회 병목과 쓰기 증폭의 공통 원인은 조회 책임과 가중치 반영 책임이 모두 `ratings` 계층에 몰려 있다는 점이었습니다. 원본 평점과 차트 계층 사이에 릴리즈 단위 결과를 따로 관리하는 평점 집계 계층이 필요했습니다.

---

## 2. 판단 기준

조회만 빠르게 만드는 것으로는 충분하지 않았습니다. 평균 평점 조회와 가중치 반영 책임이 어디까지 직접 쓰기로 이어지는지부터 다시 나눠야 했습니다. `ratings`는 원본과 변경 이력을 보존하는 기준 데이터로 남기고, 사용자에게 노출되는 릴리즈 단위 결과는 `release_rating_summary`가 맡도록 방향을 정했습니다. 원본 저장은 짧게 끝내고, 평점 집계 계층은 커밋 이후 증분 반영과 전체 재집계 보정으로 결과적 일관성에 수렴시키는 쪽이 이 서비스에 더 맞았습니다.

기준은 세 가지였습니다.

- 평점 저장 요청은 원본 `ratings` 반영까지만 직접 책임집니다.
- 조회와 차트가 함께 읽는 릴리즈 단위 결과는 `release_rating_summary`가 맡고, 커밋 이후 증분 반영과 전체 재집계 보정으로 결과적 일관성에 수렴시킵니다.
- 유저 가중치 변경은 과거 원본 평점 재기록 대신 `release_rating_summary` 재계산 경로로 넘깁니다.

### 2-1. 최종 구조

| 계층 | 책임 |
| --- | --- |
| 원본 평점 계층 (`ratings`) | 원본 평점과 변경 이력을 보존하는 기준 데이터 |
| 평점 집계 계층 (`release_rating_summary`) | 릴리즈 단위 집계 결과 보존 및 조회·차트가 함께 읽는 기준 결과 제공 |
| 차트 계층 (`chart_scores`) | 평점 집계 결과를 바탕으로 공개용 차트 점수와 순위 결과 생성 |

---

## 3. 구현 과정

### 3-1. 조회 핫패스 분리 — 그리고 늘어난 동기 쓰기 비용

먼저 릴리즈별 집계값을 `release_rating_summary`로 분리하고, 조회 경로가 원본 `ratings` 대신 이 평점 집계 계층을 읽도록 바꿨습니다.

이 전환으로 릴리즈 상세와 목록에서 평균 평점을 가져오는 비용은 크게 줄었습니다. 페이지를 열 때마다 원본 평점 전체를 다시 집계하는 대신 이미 정리된 집계 결과를 읽게 됐기 때문입니다. 실측에서도 조회 응답 시간은 806ms → 20ms로 떨어졌습니다.

하지만 처음에는 평점 생성/수정/삭제 트랜잭션 안에서 평점 집계 계층까지 동기 갱신했기 때문에, 같은 릴리즈에 쓰기가 몰릴수록 평점 집계 계층 행이 새로운 경합 지점이 됐습니다. 조회는 빨라졌는데, 평점 하나를 남길 때 응답이 오히려 무거워졌습니다. 로컬 비교에서 단순 삽입만 하는 경로는 7ms였지만, 평점 집계 계층 증분 갱신을 같은 요청 안에 붙이면 126ms로 늘었습니다.

### 3-2. 쓰기 경로와 집계 갱신 분리 — 커밋 이후 반영

1. `RatingService`가 원본 `ratings`를 저장하거나 수정합니다.
2. 같은 트랜잭션 안에서 평점 집계 계층을 건드리지 않고, 평점 이벤트만 발행합니다.
3. `RatingMessageProducer`가 `@TransactionalEventListener(phase = AFTER_COMMIT)`로 커밋 이후에만 RabbitMQ로 메시지를 보냅니다.
4. `RatingSummaryConsumer`가 릴리즈 단위 증분 값을 받아 `release_rating_summary`를 증분 갱신합니다.

원본 쓰기와 `release_rating_summary` 반영 경계는 요청 경로 밖으로 밀어냈습니다.

필요했던 건 정확히 한 번 전달이 아니라, 커밋 이후 전달 경계와 실패를 분리해 다룰 수 있는 운영 장치였습니다. RabbitMQ는 수동 `ack`, 재시도, DLQ로 증분 반영 실패를 요청 경로 밖에서 분리할 수 있었습니다.

동일 릴리즈 100건 동시 등록 평균 응답은 동기 갱신 때 126ms였지만, 커밋 이후 비동기 반영으로 전환한 뒤 12.95ms로 줄었습니다.

### 3-3. 전체 재집계 보정 경로 — 증분 반영만으로는 충분하지 않습니다

RabbitMQ의 At-Least-Once 전달 특성상, 중복 전달, 지연 재전달, 소비 실패 재시도는 구조적으로 열려 있습니다. 증분 반영만 두면 집계값이 틀어졌을 때 원본 기준으로 다시 맞출 경로가 비어 있었습니다.

`release_rating_summary`는 원본 기준 전체 재집계로 언제든 다시 만들 수 있게 두었습니다. `AntiEntropyBatchJob`은 릴리즈 id를 청크로 순회하면서 `ratings JOIN users`를 다시 읽고 모든 집계 컬럼을 한 번에 덮어씁니다. 평점 집계 계층을 다시 읽어 스스로 고치는 방식이 아니라, 원본 `ratings`와 최신 `users.weighting_score`를 기준으로 재구성합니다.

평점 생성·수정·삭제는 이벤트 증분 반영으로 빠르게 따라가고, 유저 가중치 변경과 메시지 유실 보정은 전체 재집계 경로가 맡습니다. 증분 경로와 전체 재집계 경로가 서로 덮어쓰지 않도록 `batch_synced_at` 경계도 뒀습니다. 실시간 증분 갱신 쿼리는 `event_ts > batch_synced_at` 조건에서만 반영되므로, 전체 재집계 직후 늦게 도착한 오래된 메시지가 평점 집계 계층을 다시 오염시키지 않습니다.

### 3-4. 유저 가중치 변경 흡수 — 과거 원본 평점 행을 건드리지 않습니다

유저 가중치 배치는 `users.weighting_score`와 `user_weight_stats`만 갱신합니다. 이후 반영은 과거 원본 평점 행을 다시 쓰는 방식이 아니라, 집계 경로가 최신 `users.weighting_score`를 읽어 `release_rating_summary`를 재계산하는 방식으로 처리합니다.

```
이전: 가중치 변경 → 원본 평점 행 재기록
이후: 가중치 변경 → 평점 집계 계층 재계산 → 차트 계층 (`chart_scores`) 반영
```

## 4. 결과

### 측정 조건

- 조회 비교: Spring Boot + Testcontainers MySQL 8.0 환경, 동일 릴리즈 1건에 유저 10,000명 / 평점 10,000건
- 동기 쓰기 지연: `ratings` 삽입만 수행하는 경로 vs. 같은 요청 안에서 `release_rating_summary` 증분 갱신까지 수행하는 경로
- 비동기 전환 비교: 동일 릴리즈에 100건 평점 등록 요청 동시 발행, 평균 응답 시간 측정
- 모든 수치는 로컬 포트폴리오 환경 기준이며, 운영 RPS나 분산 환경을 대표하지 않습니다.

### 수치 요약

| 항목 | 변경 전 | 변경 후 | 설명 |
| --- | --- | --- | --- |
| 릴리즈 조회 응답 | 806ms | 20ms | 원본 전수 집계 → 평점 집계 계층 단순 조회 |
| 단건 쓰기 응답 | 7ms | 126ms | 평점 집계 계층 동기 갱신 추가 시 지연 발생 (중간 단계) |
| 동시 100건 등록 평균 응답 | 126ms | 12.95ms | 평점 집계 계층 갱신을 비동기로 분리한 효과 |

### 구조적으로 달라진 것

| 비교 항목 | 이전 | 이후 |
| --- | --- | --- |
| 평점 등록 요청의 직접 쓰기 범위 | `ratings` + `release_rating_summary` 동기 갱신 | `ratings`만; 평점 집계 계층은 `AFTER_COMMIT` 이후 소비자가 반영 |
| 가중치 1회 변경 시 직접 쓰기 범위 | `users` + 과거 `ratings.weighted_score` 다수 행 | `users.weighting_score`, `user_weight_stats`만; 평점 집계 계층은 이후 집계 경로가 재계산 |
| 평점 집계 계층 정합성 회복 방식 | 증분 반영 결과에 의존 | `ratings JOIN users` 전체 재집계로 언제든 덮어쓰기 가능 |
| 차트 계층 (`chart_scores`) 입력 | 원본 `ratings` 또는 비정규화 전제 | `release_rating_summary`를 고정된 평점 집계 계층으로 사용 |

## 5. 트레이드오프와 한계

- **강한 실시간 일관성은 보장하지 않습니다.** 평점 집계 계층은 커밋 직후 일시적으로 과거 집계값을 반환할 수 있습니다.
- **Outbox 패턴을 적용하지 않았습니다.** 커밋 이후 메시지 발행 보장은 RabbitMQ 운영 장치와 전체 재집계 보정에 기댑니다. 프로세스 재시작 시 발행 누락 가능성이 남습니다.
