# 평점 조회 병목과 통계 갱신 병목을 함께 푼 구조 재설계

원본 `ratings` 직접 집계로 느려진 조회와, 유저 가중치 변경이 과거 평점 재기록으로 번지던 쓰기 문제는 별개가 아니라 같은 경계 문제였습니다. 이 문서는 `release_rating_summary`를 단순 조회용 보조 테이블이 아닌 원본 평점과 차트 계층 사이의 요약 계층으로 세우고, 읽기와 쓰기 책임을 다시 나눈 과정을 정리합니다.

- **상황**: 릴리즈 평균 평점 조회는 원본 집계에 묶여 있었고, 가중치 반영값을 평점 행에 보관하면 정책 변경 1회가 다수 행 UPDATE로 번졌습니다.
- **행동**: 평점 저장은 원본 반영까지만 맡기고, 요약 갱신은 `AFTER_COMMIT + RabbitMQ`와 전체 재집계 보정으로 비동기 수렴하도록 분리했습니다.
- **결과**: 조회 핫패스는 가벼워졌고, 가중치 변경도 과거 원본 평점 재기록 대신 요약 계층 재계산으로 흡수되도록 바뀌었습니다.

---

## 1. 문제 상황

초기 구조에서는 릴리즈 상세와 목록에서 평균 평점과 평점 수를 보여주기 위해 원본 `ratings`를 직접 읽고 집계했습니다. 평균 평점은 단순 부가 정보가 아니라 사용자가 어떤 앨범을 더 볼지 판단하는 공개 지표에 가깝습니다. 이 방식은 평점이 쌓일수록 조회 비용이 선형으로 증가했고, 조회 핫패스가 원본 테이블 스캔에 계속 묶여 있었습니다.

더 큰 문제는 읽기 비용을 줄이려는 과정에서, 유저 가중치가 반영된 값을 원본 평점 행에 미리 보관하려는 발상이 열린다는 점이었습니다. `ratings.weighted_score` 같은 비정규화 컬럼 전제를 유지하면, 조회 비용을 줄이는 대신 쓰기 문제를 더 크게 열게 됩니다.

이 전제를 유지하는 한, 유저 가중치 변경은 아래 비용으로 전파될 수밖에 없었습니다.

- 가중치 1회 변경 → 해당 유저의 과거 원본 평점 행 다수 재기록
- 대규모 UPDATE에 따른 쓰기 비용과 락 부담 증가
- 이후 집계와 차트 계산 비용까지 함께 악화

결국 조회 문제와 쓰기 증폭 문제를 동시에 끊으려면, 원본 평점과 차트 계층 사이에 별도의 요약 계층이 필요했습니다.

---

## 2. 판단 기준

### 2-1. 읽기와 쓰기를 같은 경로에서 해결하려던 시도들

- **`ratings`에 가중 반영값을 유지하면서 조회 비용을 줄이는 방식**: 조회만 보면 단순해 보였지만, 유저 가중치가 바뀔 때마다 과거 원본 평점 행을 다시 써야 했습니다. 한 번의 정책 변경이 원본 데이터를 직접 건드리는 범위를 넓히는 구조는 오래 끌고 가기 어렵다고 판단했습니다.
- **요약 계층까지 같은 트랜잭션 안에서 갱신하는 방식**: 단순해 보였지만, 사용자가 기대하는 것은 평점이 저장됐다는 빠른 응답이었습니다. 요약 계층 갱신을 같은 트랜잭션에 묶으면 쓰기 경합이 API 응답 시간으로 직접 전파됐습니다.

### 2-2. 판단 기준

`ratings`는 원본과 변경 이력을 보존하는 기준 데이터로 남기고, 사용자에게 노출되는 평균 평점과 차트용 요약값은 `release_rating_summary`가 맡도록 방향을 정했습니다. 원본 저장을 짧게 끝내고 요약 계층은 뒤에서 수렴시키는 쪽이 이 서비스에 더 맞는다고 봤습니다.

### 2-3. 최종 구조

| 계층 | 책임 |
| --- | --- |
| `ratings` | 원본 평점과 변경 이력을 보존하는 기준 데이터 |
| `release_rating_summary` | 조회 경로와 차트 배치가 함께 읽는 요약 계층. `total_rating_count`, `average_score`, `weighted_score_sum`, `weighted_count_sum`, `batch_synced_at` 관리 |
| 차트 계층 | `release_rating_summary`를 읽어 차트용 집계값 계산 |

이렇게 나누면 유저 가중치가 바뀌어도 과거 원본 평점 행을 다시 쓰지 않고, 이후 집계 경로가 최신 `users.weighting_score`를 읽어 요약 계층을 재계산하도록 만들 수 있습니다.

## 3. 구현 과정

### 3-1. 조회 핫패스 분리 — 그리고 드러난 동기 쓰기 병목

먼저 릴리즈별 집계값을 `release_rating_summary`로 분리하고, 조회 경로가 원본 `ratings` 대신 이 요약 계층을 읽도록 바꿨습니다.

이 전환으로 릴리즈 상세와 목록에서 평균 평점을 가져오는 비용은 크게 줄었습니다. 페이지를 열 때마다 원본 평점 전체를 다시 집계하는 대신 이미 정리된 집계 결과를 읽게 됐기 때문입니다. 실측에서도 조회 응답 시간은 806ms → 20ms로 떨어졌습니다.

하지만 처음에는 평점 생성/수정/삭제 트랜잭션 안에서 요약 계층까지 동기 갱신했기 때문에, 같은 릴리즈에 쓰기가 몰릴수록 요약 계층 행이 새로운 경합 지점이 됐습니다. 조회는 빨라졌는데, 평점 하나를 남길 때 응답이 오히려 무거워졌습니다. 로컬 비교에서 단순 삽입만 하는 경로는 7ms였지만, 요약 계층 증분 갱신을 같은 요청 안에 붙이면 126ms로 늘었습니다.

### 3-2. 쓰기 경로와 집계 갱신 분리 — 커밋 이후 반영

구현은 `AFTER_COMMIT + RabbitMQ` 조합으로 아래 흐름을 따릅니다.

1. `RatingService`가 원본 `ratings`를 저장하거나 수정합니다.
2. 같은 트랜잭션 안에서 요약 계층을 건드리지 않고, 평점 이벤트만 발행합니다.
3. `RatingMessageProducer`가 `@TransactionalEventListener(phase = AFTER_COMMIT)`로 커밋 이후에만 RabbitMQ로 메시지를 보냅니다.
4. `RatingSummaryConsumer`가 릴리즈 단위 증분 값을 받아 `release_rating_summary`를 증분 갱신합니다.

핵심은 RabbitMQ 자체보다, 원본 쓰기와 요약 계층 반영의 경계를 요청 경로 밖으로 밀어냈다는 점입니다.

메시지 큐 선택지는 아래처럼 검토했습니다.

| 선택지 | 맞는 점 | 선택하지 않은 이유 |
| --- | --- | --- |
| 인메모리 async | 구현이 가장 단순합니다. | 프로세스 재시작 시 작업 유실 위험이 있고, ack/retry/DLQ를 별도로 구성해야 합니다. |
| Redis 기반 큐 | 저지연 전달에 유리합니다. | 이번 문제는 캐시 재사용보다 ack/retry/DLQ 중심의 작업 큐 운영성이 더 중요했습니다. |
| Kafka | replay, 장기 보관, 다중 consumer 확장에 강합니다. | 단일 파생 집계 갱신 경계를 설명하기에는 운영 면적이 컸습니다. |
| **RabbitMQ** | 커밋 이후 전달 경계와 ack/retry/DLQ를 바로 가져갈 수 있습니다. | — |

운영 측면에서는 수동 확인 응답으로 성공/실패를 명시적으로 처리하고, 영구 실패는 DLQ로 분리해 쓰기 경로 전체가 오염되지 않도록 했습니다.

동일 릴리즈 100건 동시 등록 평균 응답은 동기 갱신 때 126ms였지만, 커밋 이후 비동기 반영으로 전환한 뒤 12.95ms로 줄었습니다.

### 3-3. 전체 재집계 보정 경로 — 증분 반영만으로는 충분하지 않습니다

RabbitMQ의 At-Least-Once 전달 특성상, 중복 전달, 지연 재전달, 소비 실패 재시도는 구조적으로 열려 있습니다. 증분 반영만 두면 "빠르게 반영된다"는 장점은 있지만, "틀어졌을 때 어떻게 되돌리나"는 답이 비어 있었습니다.

그래서 `release_rating_summary`는 원본 기준 전체 재집계로 언제든 다시 만들 수 있는 요약 계층으로 설계했습니다. `AntiEntropyBatchJob`은 릴리즈 id를 청크로 순회하면서 `ratings JOIN users`를 다시 읽고 모든 집계 컬럼을 한 번에 덮어씁니다. 이 배치는 요약 계층을 다시 읽어 스스로 고치는 것이 아니라, 원본 `ratings`와 최신 `users.weighting_score`를 기준으로 재구성합니다.

증분 경로와 전체 재집계 경로가 서로 덮어쓰지 않도록 `batch_synced_at` 경계도 뒀습니다. 실시간 증분 갱신 쿼리는 `event_ts > batch_synced_at` 조건에서만 반영되므로, 전체 재집계 직후 늦게 도착한 오래된 메시지가 요약 계층을 다시 오염시키지 않습니다.

### 3-4. 유저 가중치 변경 흡수 — 과거 원본 평점 행을 건드리지 않습니다

유저 가중치 배치는 `users.weighting_score`와 `user_weight_stats`만 갱신합니다. 이후 반영은 과거 원본 평점 행을 다시 쓰는 방식이 아니라, 집계 경로가 최신 `users.weighting_score`를 읽어 `release_rating_summary`를 재계산하는 방식으로 처리합니다.

```
이전: 가중치 변경 → 원본 평점 행 재기록
이후: 가중치 변경 → 요약 계층 재계산 → 차트 계층 반영
```

## 4. 결과

### 측정 조건

- 조회 비교: Spring Boot + Testcontainers MySQL 8.0 환경, 동일 릴리즈 1건에 유저 10,000명 / 평점 10,000건
- 동기 쓰기 지연: `ratings` 삽입만 수행하는 경로 vs. 같은 요청 안에서 `release_rating_summary` 증분 갱신까지 수행하는 경로
- 비동기 전환 비교: 동일 릴리즈에 100건 평점 등록 요청 동시 발행, 평균 응답 시간 측정
- 모든 수치는 로컬 포트폴리오 환경 기준이며, 운영 RPS나 분산 환경을 대표하지 않습니다.

### 수치 요약

| 항목 | 변경 전 | 변경 후 | 설명 |
| --- | --- | --- | --- |
| 릴리즈 조회 응답 | 806ms | 20ms | 원본 전수 집계 → 요약 계층 단순 조회 |
| 단건 쓰기 응답 | 7ms | 126ms | 요약 계층 동기 갱신 추가 시 지연 발생 (중간 단계) |
| 동시 100건 등록 평균 응답 | 126ms | 12.95ms | 요약 계층 갱신을 비동기로 분리한 효과 |

### 구조적으로 달라진 것

| 비교 항목 | 이전 | 이후 |
| --- | --- | --- |
| 평점 등록 요청의 직접 쓰기 범위 | `ratings` + `release_rating_summary` 동기 갱신 | `ratings`만; 요약 계층은 `AFTER_COMMIT` 이후 소비자가 반영 |
| 가중치 1회 변경 시 직접 쓰기 범위 | `users` + 과거 `ratings.weighted_score` 다수 행 | `users.weighting_score`, `user_weight_stats`만; 요약 계층은 이후 집계 경로가 재계산 |
| 요약 계층 정합성 회복 방식 | 증분 반영 결과에 의존 | `ratings JOIN users` 전체 재집계로 언제든 덮어쓰기 가능 |
| 차트 계층 입력 | 원본 `ratings` 또는 비정규화 전제 | `release_rating_summary`를 고정된 요약 계층으로 사용 |

## 5. 남겨둔 것

평점 등록 응답을 우선하고 요약 계층을 뒤에서 수렴시키기로 한 설계 방향상, 아래 한계는 남습니다.

- **강한 실시간 일관성은 보장하지 않습니다.** 요약 계층은 커밋 직후 일시적으로 과거 집계값을 반환할 수 있습니다.
- **Outbox 패턴을 적용하지 않았습니다.** 커밋 이후 메시지 발행 보장은 RabbitMQ 운영 장치와 전체 재집계 보정에 기댑니다. 프로세스 재시작 시 발행 누락 가능성이 남습니다.
- **수렴 보장을 정량으로 검증하지 않았습니다.** `event_ts > batch_synced_at` 조건과 전체 재집계 보정이 장기적으로 항상 수렴하는지는 실측으로 확인하지 못했습니다.
- **요약 계층 운영 비용이 추가됩니다.** 전체 재집계 배치를 지속적으로 운영해야 하므로, DB 부하와 배치 주기 조정이 필요합니다.
- **성능 수치는 로컬 환경 기준입니다.** 운영 트래픽에서 동일한 개선 비율을 기대하기 어렵습니다.

이 제약은 쓰기 경로를 짧게 유지하고 복구 경계를 나누기 위해 감수한 것입니다. 원본 저장·요약 계층·차트 계층이 나뉜 덕분에, 어느 한 계층에서 문제가 생겨도 다른 계층으로 전파되지 않습니다.
