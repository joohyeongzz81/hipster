# 차트 API를 읽기 모델·캐시·검색 경로로 다시 나눈 과정

차트 API는 사용자가 장르, 연도, 타입 같은 조건을 바꿔 가며 결과를 보고, 그 결과를 `bayesian_score DESC`로 정렬해 보여주고, `lastUpdated`까지 함께 내려주는 API입니다. 상위권 몇 개를 고정해서 보여주는 목록 조회보다, 정렬된 필터 결과를 빠르게 보여주는 쪽에 가깝습니다.

병목은 여러 곳에 겹쳐 있었습니다. `chart_scores` 조회만 느린 게 아니라 릴리즈·분류 정보 보강, 캐시 미스, 검색, 갱신 시각 조회가 한 응답 안에 같이 묶여 있었습니다. 그래서 비정규화, MySQL 최적화, Redis 응답 캐시, Elasticsearch 검색, 메타데이터 분리를 순서대로 적용하면서 병목이 어디서 어디로 옮겨 가는지 확인했습니다. 느린 쿼리 하나를 고친 작업이 아니라, 응답 경로 전체를 다시 나눈 작업이었습니다.

모든 수치는 운영 환경이 아니라 로컬 합성 데이터 벤치마크 기준입니다.

## 요약 성과

- 500만 건 합성 데이터 기준 장르 필터 미스 경로: **65,421ms → 178.37ms**
- 500만 건 합성 데이터 기준 장르 필터 반복 요청 적중 경로: **11,386ms → 16.73ms**
- 500만 건 합성 데이터 기준 갱신 시각 조회: **4,732ms → 1.00ms**

## 1. 문제 상황

차트는 사용자가 장르·연도·타입을 바꿔 가며 음악을 탐색하는 공개 API입니다. 한 릴리즈를 듣고 비슷한 장르 차트로 넘어가거나 특정 연도 상위권을 이어서 훑는 흐름이 자연스럽게 이어지기 때문에, 중요한 것은 조회 1회의 속도보다 탐색 흐름이 끊기지 않는 것입니다.

`lastUpdated`는 이 결과를 지금 기준으로 믿어도 되는지 보여 주는 값입니다. 응답이 느리거나 갱신 시각 조회가 흔들리면 탐색 경험뿐 아니라 공개 지표의 신뢰도도 같이 흔들립니다.

구현 관점에서는 `chart_scores`를 기준으로 정렬하되, 필터링과 응답 조립을 위해 릴리즈 기본 정보, 아티스트, 장르·디스크립터·언어 같은 분류 정보를 함께 읽거나 조인해야 했습니다.

- 차트는 `bayesian_score DESC` 기준으로 필터 결과를 정렬해 보여주는 API입니다.
- 차트 조회는 정렬, 필터, 응답 조립이 한 요청 안에 겹쳐 있어 병목이 한 군데에만 모이지 않았습니다.
- 반복 요청이 많아 캐시 적중 경로가 중요했는데, 캐시 미스 경로 자체가 너무 비쌌습니다.
- Redis나 Elasticsearch가 흔들리는 순간 API 전체가 같이 멈추는 구조는 피해야 했습니다.
- 검색이 빨라져도 갱신 시각 같은 공통 메타데이터 조회가 느리면 API 전체는 여전히 느렸습니다.

문제는 특정 쿼리 하나보다 응답 경로 전체에 병목이 겹쳐 있었다는 점입니다. `chart_scores` 조회, 릴리즈·분류 정보 보강, 캐시 미스, 메타데이터 조회가 한 덩어리로 묶여 있으면 무엇을 줄여야 하는지 잘 보이지 않았습니다.

## 2. 설계 방향

설계의 중심은 `chart_scores`를 기준 조회 테이블로 두고, 그 위에 겹쳐 있던 병목을 분리하는 것이었습니다.

- `chart_scores`
  - 배치가 미리 만들어 둔 기준 조회 테이블입니다.
- Redis 응답 캐시
  - 최종 응답 전체를 캐시해 반복 요청을 최전선에서 흡수합니다.
- Elasticsearch
  - 캐시 미스 시 필터 + 정렬 조건으로 릴리즈 ID 후보를 빠르게 찾는 검색 계층입니다.
- MySQL 폴백 경로
  - Redis나 Elasticsearch 장애 시 `chart_scores` 기반 동적 조회로 API를 살리는 폴백 경로입니다.
- 메타데이터 경로
  - 갱신 시각 같은 공통 메타데이터를 별도 경로로 분리해 메타데이터 병목이 API 전체를 가리지 않도록 처리합니다.

이 구조에서 Redis와 Elasticsearch는 빠른 기본 경로이고, MySQL은 장애나 예외 상황에서 API를 끊기지 않게 하는 안전망입니다. `lastUpdated`도 검색과 별개로 응답 시간을 잡아먹을 수 있었기 때문에, 메타데이터 경로로 따로 떼어 관리했습니다.

한 요청 안에 모든 일을 몰아넣지 않는 쪽으로 구조를 바꿨습니다. 정렬과 필터에 필요한 값은 `chart_scores`에 모으고, 반복 요청은 캐시가 흡수하고, 다중값 필터 검색은 검색 레이어로 빼고, 공통 메타데이터는 따로 읽도록 나눴습니다.

## 3. 해결 과정

수치는 모두 로컬 합성 데이터 벤치마크 기준입니다. 절대 성능보다, 같은 API와 같은 데이터셋에서 병목이 어디로 이동했고 각 단계가 응답 시간에 어떤 영향을 줬는지 비교했습니다. 측정 환경은 문서 끝 부록에 정리했습니다.

본문에는 대표 시나리오만 싣고, 전체 시나리오 목록과 측정표는 부록에 실었습니다. 별도 표기가 없으면 모두 `page=0`, `size=20`, 같은 응답 DTO 기준입니다.

| 시나리오 | 조건 | 의도 |
| --- | --- | --- |
| `S1_BASELINE` | 필터 없음 | 기본 조회 경로 |
| `S2_RELEASE_TYPE_ALBUM` | `releaseType=ALBUM` | 단일값 필터 최적화 효과 |
| `S4_GENRE_1` | `genreIds=[1]` | JSON 필터 병목 대표 |
| `G2_GENRES_1_7` | `genreIds=[1,7]` | 복합 JSON 필터 대표 |

### 3-0. 지표 읽는 법

| 용어 | 뜻 |
| --- | --- |
| `*Millis` | 단일 실행 1회에서 서버 내부 단계를 계측한 시간(ms)입니다. |
| `*Avg` | 같은 시나리오를 여러 번 반복 실행했을 때의 평균 시간(ms)입니다. |
| `wallMillis`, `wallAvg` | 요청 시작부터 응답 수신까지의 실제 경과 시간입니다. 실제 API 체감에 가장 가깝습니다. |
| `totalMillis`, `totalAvg` | 검색, 메타데이터 조회, 응답 조립 등 서버 내부 처리 전체를 합산한 시간입니다. |
| `searchMillis`, `searchAvg` | 필터 + 정렬 조건으로 릴리즈 ID 후보를 찾는 단계의 시간입니다. |
| `hydrateMillis`, `hydrateAvg` | 찾은 릴리즈 ID로 `chart_scores`와 릴리즈를 다시 읽고, 검색 결과 순서대로 복원하는 복원 단계(`hydrate`)의 시간입니다. |
| `lastUpdatedMillis`, `lastUpdatedAvg` | 공통 메타데이터인 갱신 시각(`lastUpdated`)을 읽는 시간입니다. |
| `assembleMillis`, `assembleAvg` | 최종 응답을 조립하고 아티스트 이름을 채우는 응답 조립 단계의 시간입니다. |
| `cold miss` | 관련 캐시를 비운 뒤 처음 수행한 최초 미스 요청입니다. |
| `miss` | 해당 캐시 키가 비어 있어 검색과 응답 조립을 모두 거친 미스 요청입니다. |
| `hit` | 캐시에 저장된 응답을 바로 반환한 적중 요청입니다. |
| `Handler_read_key` | 인덱스 키를 통해 읽은 횟수입니다. |
| `Handler_read_next` | 인덱스에서 다음 레코드를 순차적으로 읽은 횟수입니다. |
| `Handler_read_rnd_next` | 랜덤/풀스캔 성격의 읽기 횟수입니다. |
| `Sort_merge_passes` | 정렬 과정에서 merge pass가 발생한 횟수입니다. |

### 3-1. `chart_scores`를 차트 조회용 projection으로 확장했습니다

처음에는 조회 모델부터 바꿨습니다. 원래 차트 조회는 릴리즈를 중심으로 장르·디스크립터·언어 같은 N:M 관계를 계속 따라가며 조건을 평가하는 구조였고, 차트 결과도 사실상 "필터를 조합한 뒤 `bayesian_score DESC`로 정렬한 검색 결과"였습니다. 이 구조에서는 필터 한 번, 정렬 한 번마다 JOIN 비용을 매 요청마다 다시 지불해야 했습니다.

그래서 `chart_scores`를 단순 점수 테이블이 아니라 차트 전용 projection으로 확장했습니다. 배치에서 차트 입력 데이터를 읽을 때 `releaseType`, `releaseYear`, `locationId`, `genreIds`, `descriptorIds`, `languages`를 같이 만들어 `chart_scores`에 넣고, 화면 조립에만 필요한 릴리즈 제목, 발매일, 아티스트 이름은 조회 시점의 `hydrate` 단계로 남겼습니다. 장르·디스크립터·언어도 배치 시점에 `JSON_ARRAYAGG`로 묶어 `chart_scores`에 저장했습니다.

이전에는 조회 시점에 JOIN으로 조합하던 값을, 이후에는 배치 시점에 projection 한 행으로 미리 만들어 두고 조회 시점에는 그 행을 정렬과 필터에 바로 쓰도록 바꿨습니다. JOIN 책임을 read path에서 batch path로 옮긴 셈입니다.

| 시나리오 | CH1 정규화 조인 | CH2 비정규화 기준 조회 테이블 | 변화 |
| --- | --- | --- | --- |
| `S1_BASELINE` | `85,652ms` | `9,303ms` | 무필터 기본 경로 개선 |
| `S2_RELEASE_TYPE_ALBUM` | `54,488ms` | `8,864ms` | 단일값 필터 개선 |
| `S4_GENRE_1` | `65,421ms` | `12,034ms` | 장르 필터 개선 |
| `G2_GENRES_1_7` | `61,634ms` | `14,089ms` | 멀티 장르 필터 개선 |

- 정규화 JOIN 병목은 확실히 줄었습니다.
- 특히 "릴리즈 + 장르 + 디스크립터 + 언어"를 매번 따라가던 읽기 비용이 `chart_scores` 한 테이블 쪽으로 모였습니다.
- 다만 여기서 줄어든 것은 JOIN 비용이었고, 정렬과 JSON 필터 비용은 그대로 남았습니다.

MySQL 지표도 같은 방향을 보여줬습니다. `S4_GENRE_1` 기준으로 조인 관련 읽기는 크게 줄었습니다.

- `Handler_read_key`: `15,920,719 -> 67`
- `Handler_read_next`: `8,955,317 -> 3`

반면 실행 계획에서는 여전히 `chart_scores`를 크게 읽고 정렬하는 성격이 남았습니다. 이 단계만으로는 `bayesian_score DESC` 정렬 비용을 제거하지 못했습니다.

- `Handler_read_rnd_next`: `14,633,960 -> 10,001,006`
- `Sort_merge_passes`: `592 -> 699`

이 단계에서 "정규화 테이블을 계속 JOIN하는 구조"는 "projection 한 행을 먼저 읽는 구조"로 바뀌었습니다. 하지만 `JSON_CONTAINS` 평가와 정렬 비용 때문에 여전히 `chart_scores`를 넓게 읽는 성격이 남았습니다.

**이 단계의 결론:** 조인 병목은 줄였지만, JSON 필터와 정렬 비용은 여전히 주 병목으로 남았습니다.

### 3-2. 인덱스로 MySQL 조회 경로를 더 줄였습니다

두 번째 단계에서는 비정규화한 `chart_scores`를 MySQL 안에서 어디까지 빠르게 읽을 수 있는지 확인했습니다.

이전 쿼리 패턴은 Repository의 native query처럼 `:param IS NULL OR ...` 조건으로 동적 필터를 우회하는 방식이었습니다.

```sql
WHERE (:genreId IS NULL OR JSON_CONTAINS(c.genre_ids, JSON_OBJECT('id', :genreId)))
AND (:locationId IS NULL OR c.location_id = :locationId)
AND (:year IS NULL OR c.release_year = :year)
AND (:releaseType IS NULL OR c.release_type = :releaseType)
ORDER BY c.bayesian_score DESC
```

이 패턴은 쿼리 시점의 실제 필터 조합이 흐려져 인덱스를 적극적으로 타기 어려웠습니다. 그래서 `idx_chart_scores_filter_sort`를 기준으로 `release_type`, `release_year`, `location_id`, `bayesian_score` 조합에서 단일값 필터와 정렬을 최대한 한 경로로 묶도록 다듬었습니다.

효과가 있었던 이유는, 이 API의 비용이 "후보를 많이 읽는 비용"과 "읽은 뒤 다시 정렬하는 비용"으로 나뉘어 있었기 때문입니다. 단일값 조건이 인덱스 경로에 걸리면 MySQL은 후보를 먼저 줄일 수 있고, 같은 경로에서 `bayesian_score DESC`까지 활용되는 경우에는 별도 `filesort` 없이 상위 결과를 바로 가져올 수 있습니다. 인덱스가 잘 맞는 시나리오에서는 필터와 정렬이 한 번에 처리되면서 응답 시간이 같이 줄어듭니다.

최대 필터 시나리오에서는 `Using index condition; Using where`까지 내려갔고, 디스크 기반 정렬도 제거됐습니다. 반대로 선행 컬럼이 비어 있거나 JSON 조건이 중심인 시나리오에서는 이 이점이 거의 사라졌습니다.

| 시나리오 | CH2 비정규화 기준 조회 테이블 | CH3 쿼리 재구성 + 인덱스 | 변화 |
| --- | --- | --- | --- |
| `S1_BASELINE` | `9,303ms` | `6,604ms` | 무필터 경로 추가 개선 |
| `S2_RELEASE_TYPE_ALBUM` | `8,864ms` | `7,159ms` | 단일값 필터 개선 |
| `S4_GENRE_1` | `12,034ms` | `11,564ms` | 장르 JSON 필터는 미미한 개선 |
| `G2_GENRES_1_7` | `14,089ms` | `13,773ms` | 멀티 장르 필터는 거의 변화 없음 |

- `releaseType`, `year`, `location`처럼 B-Tree 인덱스로 다룰 수 있는 단일값 조건은 실제로 개선됐습니다.
- 후보를 먼저 줄이고 정렬 비용도 같이 줄일 수 있었기 때문입니다.
- 반면 `genreIds`, `descriptorIds`, `languages`처럼 `JSON_CONTAINS` 기반 다중값 필터는 인덱스가 직접 해결하지 못했습니다.

이 차이는 실행 계획과 DB 지표에도 그대로 드러났습니다. 단일값 복합 조건에서는 `Using index condition`이 보였지만, 선행 컬럼이 빠진 경우에는 `Using filesort`가 다시 나타났습니다. 인덱스가 맞는 조합에서는 빨라졌지만, 그렇지 않은 조합에서는 여전히 많이 읽고 다시 정렬해야 했습니다. 같은 `S4_GENRE_1` 기준으로 CH3까지 온 뒤에도

- `Handler_read_key`: `67 -> 62`
- `Handler_read_rnd_next`: `10,001,006 -> 10,001,005`
- `Sort_merge_passes`: `699 -> 650`

수준이어서, 조인 관련 읽기는 이미 줄였지만 JSON 필터와 정렬 비용은 거의 그대로 남아 있었습니다. 500만 건에 대한 `JSON_CONTAINS` 평가 자체가 남아 있었고, 이 비용은 "인덱스로 후보를 줄인 뒤 상위 몇 개만 읽는다"는 MySQL 최적화가 잘 먹히는 형태가 아니었습니다.

여기서 방향이 선명해졌습니다. 이 API는 본질적으로 "다중 필터 조합 + 점수 정렬" 문제인데, MySQL B-Tree 인덱스는 단일값 조건과 정렬에는 강하지만 JSON 배열 안의 다중값 포함 여부를 빠르게 거르는 데는 한계가 있었습니다. 그래서 다음 단계에서는 "MySQL을 버리는 것"보다 MySQL이 잘하는 단일값·정렬 경로와 못하는 다중값 검색 경로를 분리하는 데 집중했습니다.

읽기 성능이 붙는 만큼 쓰기 비용도 함께 늘었습니다. `chart_scores`를 갱신할 때 인덱스 유지 비용이 따라 붙고, 배치 재생성이나 upsert 구간에서도 그 부담을 계속 감수해야 했습니다.

**이 단계의 결론:** 단일값 필터는 더 빨라졌지만, JSON 다중값 필터는 MySQL만으로 기본 조회 경로가 되기 어려웠습니다.

### 3-3. 반복 요청은 Redis가 흡수하고, 비싼 미스 경로는 별도로 확인했습니다

세 번째 단계는 Redis 응답 캐시입니다.

차트 조회는 두 가지 특징이 있습니다.

- 같은 필터 조합과 같은 페이지를 반복해서 읽는 비율이 높습니다.
- 데이터는 요청마다 바뀌는 것이 아니라 배치 단위로 갱신됩니다.

즉 이 API는 "쓰기 빈도는 낮고 같은 읽기가 반복되는 read-heavy API"에 가깝습니다. 이런 경우에는 결과를 다시 계산하는 것보다 최종 응답 전체를 look-aside 방식으로 캐시하는 편이 훨씬 유리합니다. 실제로는 `chart:v1:{publishedVersion}:...` 형태의 키에 필터 조합과 페이지를 정규화해 직렬화했고, 같은 요청이면 항상 같은 캐시 키로 떨어지게 만들었습니다. 버전 prefix를 붙여 배치 publish 이후 이전 응답과 섞이지 않게 했습니다.

이 전략이 잘 맞았던 이유는, 이 API의 비싼 비용이 DB 한 번이 아니라 `검색 -> hydrate -> 응답 조립` 전체에 퍼져 있었기 때문입니다. 응답 캐시가 hit 되면 이 경로를 통째로 건너뛸 수 있으므로, Redis는 부분 캐시보다 최종 응답 캐시로 쓸 때 효과가 더 컸습니다. 같은 요청이 반복될수록 "다시 계산하지 않는다"는 점 자체가 가장 큰 최적화가 됐습니다.

그래도 캐시가 비어 있는 `miss` 경로는 그대로 두고, 이후 단계에서 계속 별도 측정했습니다. 캐시 hit만 보고 "빨라졌다"고 말하면, 정작 비싼 검색 경로가 어디에 남아 있는지 가려지기 때문입니다.

`최초 미스`(`cold miss`)와 `미스`(`miss`)는 `totalMillis`, `적중`(`hit`)은 `wallMillis` 기준입니다.

| 시나리오 | 최초 미스 | 미스 | 적중 |
| --- | --- | --- | --- |
| `S1_BASELINE` | `6,556ms` | `6,768ms` | `16.85ms` |
| `S4_GENRE_1` | `11,382ms` | `11,386ms` | `16.73ms` |
| `G2_GENRES_1_7` | `13,621ms` | `13,295ms` | `22.98ms` |

- 적중 경로는 15~23ms대로 떨어집니다.
- 하지만 미스 경로는 거의 그대로 남아 있습니다.

Redis는 반복 요청을 흡수하는 데 효과적이었습니다. 이 API처럼 같은 요청이 반복되고 데이터가 배치 단위로 갱신되는 구조에서는 응답 캐시가 잘 맞았고, 그 결과 hit 경로는 수십 ms대로 떨어졌습니다. 하지만 캐시가 비는 순간에는 다시 비싼 검색 경로 전체를 타야 했기 때문에, 미스 경로의 핵심 병목은 남아 있었습니다.

캐시를 빠르게 쓰는 대신 무효화와 cold start 부담도 함께 생겼습니다. 배치 직후 캐시가 비면 한동안 미스가 몰리고, 같은 요청이 한꺼번에 들어오는 구간에서는 stampede 위험도 생깁니다.

**이 단계의 결론:** Redis는 반복 요청을 흡수했지만, 캐시 미스 경로의 핵심 병목은 그대로 남았습니다.

### 3-4. Elasticsearch를 검색 경로로 전환하고, MySQL은 폴백 경로로 남겼습니다

네 번째 단계에서는 남아 있던 미스 경로 병목을 줄이기 위해 Elasticsearch를 검색 경로로 분리했습니다.

이 API의 장르·디스크립터·언어 필터는 결국 "배열 필드 안에 특정 값들이 들어 있는가"를 묻는 문제입니다. MySQL에서는 이것이 `JSON_CONTAINS` 평가로 남아 500만 건 단위 스캔과 함수 평가 비용을 만들었지만, Elasticsearch에서는 `genreIds`, `descriptorIds`, `languages`를 배열 필드로 색인해 두고 바로 걸러낼 수 있습니다. 문서도 `releaseType`, `releaseYear`, `locationId`, `genreIds`, `descriptorIds`, `languages`, `bayesianScore`를 각각 검색용 필드로 나눠 갖고 있습니다.

개선으로 이어진 이유는, 이 API의 가장 비싼 단계가 "많은 행을 읽은 뒤 JSON 함수를 반복 평가해 후보를 줄이는 과정"이었기 때문입니다. ES로 옮기면 같은 조건이 "해당 값을 가진 문서 집합을 바로 좁히는 과정"으로 바뀌므로, 검색 단계 자체가 훨씬 짧아집니다. MySQL에서는 후보를 많이 읽고 버리는 비용이 컸다면, ES에서는 후보를 훨씬 앞단에서 줄일 수 있었습니다.

이 API는 텍스트 검색이 아니라 필터 검색이기 때문에, ES에서도 scoring보다 filter가 더 중요했습니다. `genreIds`, `descriptorIds`, `languages`, `locationId`, `releaseType`, `releaseYear`를 bool filter에 넣고, 결과는 `bayesianScore` 내림차순으로 정렬해 `releaseId`만 가져오도록 구성했습니다. 그래서 ES는 최종 응답 엔진보다, 비싼 필터 검색 비용을 먼저 줄이는 엔진으로 쓸 때 가장 효과적이었습니다.

검색은 Elasticsearch가 맡고, 최종 응답 조립과 정합성 기준은 MySQL 쪽에 남겨 뒀습니다. 그래서 ES는 `releaseId` 후보를 찾고, 그 뒤에는 MySQL에서 `chart_scores`와 릴리즈를 다시 읽어 `hydrate`하고 응답을 조립했습니다. 또한 ES가 실패하거나 비어 있어도 API가 죽지 않도록 MySQL 경로를 폴백으로 남겼습니다.

여기서 중요한 점은 "Elasticsearch로 완전히 갈아탔다"가 아니라, "MySQL이 가장 비싸게 처리하던 다중값 필터 검색만 ES로 빼고, 응답 복원과 폴백은 MySQL에 남겼다"는 데 있습니다. 그 결과 미스 경로에서도 가장 먼저 줄어든 것은 검색 단계의 비용이었습니다.

이 구조로 가면서 저장소가 하나 더 늘었습니다. 이제는 MySQL projection과 Elasticsearch 색인을 함께 맞춰야 했고, 검색은 빨라졌지만 구조와 운영 복잡도는 더 커졌습니다. ES가 흔들릴 때를 대비한 MySQL 폴백도 남겨 두었지만, 그 폴백 자체는 여전히 비싼 경로였습니다.

값은 `totalMillis`, `searchMillis`, `hydrateMillis`, `lastUpdatedMillis` 기준입니다.

| 시나리오 | totalMillis | searchMillis | hydrateMillis | lastUpdatedMillis |
| --- | --- | --- | --- | --- |
| `S1_BASELINE` | `4,940ms` | `73ms` | `13ms` | `4,847ms` |
| `S2_RELEASE_TYPE_ALBUM` | `4,906ms` | `52ms` | `12ms` | `4,836ms` |
| `S4_GENRE_1` | `4,790ms` | `37ms` | `13ms` | `4,732ms` |
| `G2_GENRES_1_7` | `5,249ms` | `367ms` | `11ms` | `4,865ms` |

- Elasticsearch 검색 자체는 빨랐습니다.
- 하지만 실제 API는 여전히 4,700~5,300ms 수준이었습니다.
- "검색 병목을 옮겼는데도 왜 체감이 안 나오지?"라는 새로운 문제가 남았습니다.

이 단계까지 오면 검색 경로 자체는 분리됐습니다. 그런데도 실제 응답 시간이 거의 줄지 않았기 때문에, 이 시점부터는 최적화보다 트러블슈팅이 더 중요해졌습니다.

**이 단계의 결론:** Elasticsearch는 검색을 줄였지만, 실제 응답 시간은 다른 공통 병목에 가려져 있었습니다.

### 3-5. 트러블슈팅: Elasticsearch를 붙였는데도 왜 API는 느렸나

여기서는 새 경로를 더 붙이기보다, 왜 실제 응답이 줄지 않는지 다시 쪼개서 봤습니다. Elasticsearch를 붙인 뒤 `searchMillis`는 수십~수백 ms로 내려갔는데, `totalMillis`는 여전히 5초 안팎이었습니다. 검색이 빨라졌다는 사실과 실제 API가 빨라졌다는 사실이 바로 이어지지 않았습니다.

원인을 분해해 보니 문제는 검색이 아니라 공통 메타데이터 조회였습니다. 당시에는 각 요청마다 `chart_scores`에서 최신 `lastUpdated`를 다시 읽어 왔고, 이 경로가 검색 결과와 무관한 고정 비용으로 응답 전체를 지배하고 있었습니다. ES를 붙인 뒤 다시 보게 된 질문도 "검색 엔진이 왜 느리냐"가 아니라 "검색 밖의 어떤 공통 후처리가 응답 시간을 붙잡고 있느냐"였습니다.

그래서 해결도 검색 엔진 추가 최적화가 아니라 메타데이터 경로 분리로 갔습니다. `lastUpdated`를 `chart-meta:last-updated:v1` Redis 메타데이터 키로 분리하고, 차트 배치 완료 시 이 값을 갱신하도록 바꿨습니다. 조회 시점에는 더 이상 `chart_scores`를 다시 정렬 조회하지 않고, 메타데이터 키를 우선 읽고 필요할 때만 DB로 폴백하도록 분리했습니다.

재측정 구간은 반복 실행 평균이라 `wallAvg`, `totalAvg`, `lastUpdatedAvg`를 함께 적었습니다.

| 시나리오 | 검색 단계 전체 시간 | 당시 갱신 시각 조회 시간 | 재측정 실측 평균 | 재측정 내부 처리 평균 | 재측정 갱신 시각 평균 |
| --- | --- | --- | --- | --- | --- |
| `S2_RELEASE_TYPE_ALBUM` | `4,906ms` | `4,836ms` | `47.45ms` | `16.00ms` | `1.00ms` |
| `S4_GENRE_1` | `4,790ms` | `4,732ms` | `178.37ms` | `145.67ms` | `1.00ms` |
| `G2_GENRES_1_7` | `5,249ms` | `4,865ms` | `206.43ms` | `143.33ms` | `1.33ms` |

이 단계에서야 비로소 Elasticsearch 검색 가속 효과가 실제 API 응답 시간으로 드러났습니다. 검색보다 메타데이터가 더 느린 상태에서는 어떤 검색 최적화도 체감되지 않았고, 갱신 시각 조회를 분리한 뒤에야 미스 경로가 수백 ms대로 내려왔습니다.

이후에는 `lastUpdated`의 의미도 더 엄밀하게 관리해야 했습니다. 단순히 DB에서 바로 읽는 값이 아니라, 배치와 공개 시점을 거쳐 별도로 관리되는 메타데이터가 되었기 때문입니다.

**이 단계의 결론:** ES 도입의 진짜 마지막 병목은 검색이 아니라 공통 메타데이터 조회였고, 이를 분리한 뒤에야 검색 가속 효과가 실제 응답 시간으로 이어졌습니다.

## 5. 현재 서빙 구조

현재 구조는 Elasticsearch가 차트 API 응답을 직접 만드는 형태가 아니라, `chart_scores` 위에 Redis·Elasticsearch·MySQL 폴백 경로·메타데이터 경로를 분리한 서빙 구조입니다.

```text
/api/v1/charts?page&size
  -> 요청 조건 기준 캐시 키 계산
  -> Redis 응답 캐시 조회
  -> 캐시 적중: 응답 즉시 반환
  -> 캐시 미스:
       -> Elasticsearch에서 릴리즈 ID 후보 검색
       -> 릴리즈 ID 목록 반환
       -> chart_scores + 릴리즈 조회
       -> 아티스트 조회 + 응답 조립 (갱신 시각, 순위 목록)
       -> Redis 캐시 기록

Redis 읽기/쓰기 실패
  -> 검색/조립 경로로 계속 진행

Elasticsearch 실패
  -> MySQL 폴백 조회
  -> 동일한 조립 경로로 응답 반환

메타데이터
  -> 갱신 시각 별도 조회
```

- 차트는 요청마다 원천 데이터를 다시 계산하지 않고, 배치가 만들어 둔 `chart_scores`를 조회합니다.
- 빠른 기본 조회 경로는 Redis와 Elasticsearch이고, MySQL 폴백 경로는 검색 장애나 인덱스 미반영 상황에서도 API가 끊기지 않도록 남겨 둔 안전장치입니다.
- 갱신 시각은 검색 성공 여부와 무관한 공통 조회이므로, 별도 메타데이터 경로로 분리해 부분 실패와 병목을 격리합니다.

## 6. 남겨둔 것

- MySQL 폴백 경로는 API를 살리는 안전망이지만 여전히 비싼 경로입니다. 부분 장애 상황에서 어느 정도의 성능 저하를 허용할지까지 포함해, 고가용성 관점의 운영 기준이 더 필요합니다.
- 차트 최신성은 배치 반영 주기에 종속됩니다. 즉시 반영형 API가 아니라 배치 주기에 맞춰 갱신되는 구조입니다.
- 응답 조립 과정에서 릴리즈 조회와 아티스트 조회가 추가로 필요하므로, 검색 외 조회 비용은 계속 관리해야 합니다.
