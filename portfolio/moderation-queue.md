# 검수 적체와 담당 전환을 설명할 수 있게 만든 검수 대기열

처음에는 공식 담당 전환 경로 하나가 빠져서 불편한 정도라고 생각했습니다. 그런데 실제로 더 큰 문제는 누가 무엇을 맡고 있는지, 왜 지금 상태가 됐는지, 무엇이 늦고 있는지를 시스템이 설명하지 못한다는 점이었습니다.

검수 요청 저장소를 단순 상태 보관소가 아니라, 현재 상태·운영 이력·점유 회수·적체 해석 기준을 나눠 운영 가능한 대기열로 다시 정리했습니다.

---

## 1. 문제 상황

초기 검수 구조에도 운영자가 항목을 점유하고 승인·반려하는 최소 흐름은 있었습니다. 하지만 누가 맡고 있는지, 점유가 언제 방치됐는지, 왜 다시 대기로 돌아왔는지까지 설명할 수는 없었습니다. 이 상태로는 검수 적체를 운영 가능한 대기열처럼 다루기 어려웠습니다.

특히 운영 관점에서 불편했던 지점은 아래와 같았습니다.

- 검수 중인 항목을 공식적으로 다른 운영자에게 넘기는 경로가 없었습니다.
- 최신 상태는 보여도, 누가 언제 점유했고 왜 다시 대기로 돌아왔는지는 설명하기 어려웠습니다.
- 검수 적체와 `SLA` 초과 상태를 한눈에 읽을 수 있는 기준이 없었습니다.
- 운영자는 검수 판단뿐 아니라 방치된 항목 정리와 담당 조정까지 직접 떠안아야 했습니다.

이 문제는 단순히 API 몇 개가 빠졌다는 뜻이 아니었습니다. 시스템이 검수 적체를 운영 책임의 일부로 받아들이지 않고 있었고, 그 결과 운영자가 메신저와 기억에 의존해 대기열을 관리해야 했습니다.

방치된 점유 회수가 일부 요청 경로에 기대고 있던 구조도 있었지만, 이 문서에서 더 중요하게 본 문제는 조회 로직 한 줄이 아니라 운영 책임의 위치였습니다. 검수 적체를 줄이는 것만큼, 그 적체를 왜 그렇게 해석하는지 설명할 수 있어야 했습니다.

---

## 2. 판단 기준

중요했던 건 검수 기능을 더 늘리는 일이 아니라, 검수 적체를 어떤 기준으로 해석하고 누가 그 책임을 지는지 구조 안에 남기는 일이었습니다. 담당 전환, 현재 상태 설명, 방치된 점유 회수, `SLA` 가시성은 따로 보이지만 실제로는 같은 운영 질문에 답하는 구조여야 했습니다.

기준은 네 가지였습니다.

- 담당 전환은 새 업무 단계를 만들지 않고, 기존 검수 중 상태 안의 운영 전이로 남깁니다.
- 현재 상태 설명과 운영 이력 설명은 서로 다른 질문이므로 분리합니다.
- 검수 적체와 `SLA`는 응답과 메트릭이 같은 기준으로 읽게 합니다.
- 방치된 점유 회수는 운영자의 우연한 조회가 아니라 시스템 책임으로 분리합니다.

### 2-1. 최종 구조

| 계층 | 책임 |
| --- | --- |
| 현재 상태 계층 (`moderation_queue`) | 현재 담당자, 점유 만료 시각, 검수 상태처럼 "지금 무엇인가"를 설명 |
| 운영 이력 계층 (`moderation_audit_trail`) | 점유, 담당 전환, 만료 회수, 승인·반려처럼 "왜 이렇게 됐는가"를 설명 |
| 점유 회수 경로 (백그라운드 회수 + 마지막 액션 보정) | 방치된 점유를 요청 유입과 분리해 정리하고 마지막 액션 직전 만료 여부를 한 번 더 보정 |
| 적체 해석 기준 | 개별 항목 경과 시간, `SLA` 초과 여부, 목록 적체를 같은 기준으로 노출 |

이 구조의 목적은 검수 기능을 더하는 것이 아니라, 운영자가 "누가 맡고 있나", "왜 지금 상태가 됐나", "무엇이 늦고 있나"를 시스템 안에서 같은 기준으로 읽게 만드는 것이었습니다.

---

## 3. 구현 과정

### 3-1. 공식 담당 전환 경로를 만들고 상태 모델은 늘리지 않았습니다

담당 전환은 공식 전환 경로로 열었습니다. 현재 점유자나 관리자가 다른 운영자에게 항목을 넘길 수 있게 했고, 대상은 검수 역할을 가진 사용자로 제한했습니다.

이때 항목의 업무 단계는 그대로 검수 중으로 두고, 담당자와 점유 시각만 새 기준으로 갱신했습니다. 담당 전환은 별도 상태를 추가하지 않고 운영 이력으로만 남겼습니다. 이 방식 덕분에 상태 모델은 불필요하게 늘리지 않으면서도, 실제 운영에서 필요한 담당 전환 절차는 시스템 안에 넣을 수 있었습니다.

### 3-2. 운영 이력을 남기고 회수 충돌은 드러냈습니다

운영 이력에는 점유, 점유 해제, 점유 만료 회수, 담당 전환, 승인, 반려를 남기도록 했습니다. 이렇게 해야 "왜 지금 대기 상태인지", "왜 담당자가 바뀌었는지", "왜 반려됐는지"를 사후에 설명할 수 있습니다.

방치된 점유 회수 경로가 따로 생기면서 검수 대기열을 갱신하는 쓰기 경로도 늘었습니다. 그래서 회수 작업에는 낙관적 락을 두고 항목별 트랜잭션으로 처리했습니다. 충돌이 나면 해당 항목만 건너뛰고, 조용한 덮어쓰기보다 충돌이 드러나는 쪽을 택했습니다.

### 3-3. 응답과 메트릭이 같은 질문에 같은 답을 주게 했습니다

응답과 메트릭이 개별 항목 경과 시간, `SLA` 초과 여부, 목록 적체를 같은 기준으로 읽도록 맞췄습니다.

- 개별 항목에서는 제출 후 경과 시간과 `SLA` 초과 여부를 함께 봅니다.
- 목록 응답에서는 대기 수, 검수 중 수, `SLA` 초과 수를 같은 기준으로 집계합니다.
- 메트릭에서도 같은 기준의 backlog를 노출합니다.

이렇게 해야 운영자가 개별 항목, 목록 응답, 메트릭을 오가도 같은 질문에 같은 답을 얻습니다.

### 3-4. 서비스 흐름이 실제로 닫히는지 시나리오로 확인했습니다

서비스 테스트에서는 아래를 확인했습니다.

- 만료된 점유가 요청 유입 없이도 회수되는지
- 마지막 액션 시점에도 만료된 점유를 들고 승인하거나 담당 전환하지 못하는지

시나리오 테스트에서는 아래 운영 흐름을 확인했습니다.

- 점유 만료 -> 백그라운드 회수 -> 다른 운영자 재점유 -> 최종 승인
- 담당 전환 -> `SLA` 초과 상태 유지 -> 새 담당자 최종 반려

이렇게 해서 방치된 점유 회수, 담당 전환, `SLA` 상태 유지가 사용자 요청과 분리돼도 실제 서비스 흐름으로 닫히는지 확인했습니다.

## 4. 결과

이번 작업으로 검수 대기열은 아래처럼 달라졌습니다.

| 관점 | 이전 | 이후 |
| --- | --- | --- |
| 담당 전환 | `점유 반납 -> 재점유` 우회 절차 | 공식 담당 전환 경로 |
| 현재 상태 설명 | 최신 상태 위주 | 현재 상태 + 운영 이력 분리 |
| 검수 적체 가시성 | 누가 무엇을 맡고 무엇이 늦는지 설명 어려움 | 응답과 메트릭에서 같은 `SLA` 기준으로 확인 |
| 방치된 점유 회수 | 운영자 요청 흐름에 기대는 정리 | 백그라운드 회수 + 마지막 액션 보정 |

검수 적체를 운영자가 수동으로 감당하던 구조에서 시스템이 일부 책임지는 구조로 바뀌었습니다. 운영자는 이제 담당 전환, 적체 상태, 현재 상태의 이유를 시스템 안에서 읽을 수 있습니다.
