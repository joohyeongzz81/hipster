# 검수된 기여만 적립으로 인정하게 만든 적립 원장

처음에는 적립을 기여 유도용 포인트 기능 정도로 생각했습니다. 사용자 기반 카탈로그 서비스는 초기에 데이터가 부족하니, 승인된 제출에 보상을 붙이면 어느 정도 버틸 수 있을 것 같았기 때문입니다. 그런데 막상 흐름을 붙이기 시작하자, 보상보다 먼저 적립 기준을 정리해야 한다는 점이 더 크게 보였습니다. 무엇을 적립 대상으로 인정할지, 같은 기여가 두 번 적립되지 않는 이유, 적립이 안 된 이유를 어떻게 설명할지부터 닫아야 했습니다.

- **상황**: 서비스 초기 콜드 스타트를 버티기 위해 사용자 기여 유도 장치가 필요했지만, 기준 없이 설계한 보상은 중복·저품질 제출을 늘릴 수 있었습니다.
- **행동**: 검수(moderation)와 적립(reward)을 별개의 도메인 경계로 나누고, 승인 이벤트를 Outbox로 확정한 뒤 reward가 자기 불변식으로 적립 원장을 관리하는 구조로 설계했습니다.
- **결과**: 적립이 단순 포인트 가산이 아니라, 어떤 승인 기여가 왜 적립됐고 왜 차단됐고 왜 되돌려졌는지를 설명할 수 있는 도메인이 됐습니다.

---

## 1. 문제 상황

이 서비스는 운영자가 직접 카탈로그를 채우는 구조가 아니라, 사용자의 제출과 검수를 통해 릴리즈와 리뷰를 늘려 가는 구조입니다. 초기에 가장 먼저 부딪히는 문제가 콜드 스타트였습니다. 보여 줄 카탈로그가 부족하면 평가와 탐색이 활발해지기 어렵고, 그러면 기여도 들어오기 어려워지는 악순환이 있었습니다.

적립은 이 지점을 버티기 위한 장치였습니다. 카탈로그 공백 자체를 보상만으로 해결할 수는 없지만, 초반에 필요한 기여를 끌어오는 역할은 있었습니다. 다만 핵심은 "보상을 준다"가 아니라 "무엇을 보상으로 인정할 것인가"였습니다.

검증되지 않은 제출까지 적립 대상으로 열어 두면 구조적인 문제가 생겼습니다.

- 숫자만 맞추는 중복·저품질 제출도 보상 후보가 됩니다.
- 운영자는 품질 판단과 보상 정정을 동시에 처리해야 합니다.
- 사용자는 적립 기준을 이해하기 어려워집니다.

그래서 적립 입력은 제출이 아니라 검수 승인 결과여야 했습니다. 그런데 여기서 중요한 구분이 있었습니다. 적립은 평점 집계 계층처럼 원본 기준이 살아 있으면 다시 계산해 맞출 수 있는 파생 값이 아니었습니다. 검수와 적립은 서로 다른 도메인 경계 위에 있고, 적립 원장은 외부 이벤트를 받아 자기 경계 안에서 독립적으로 상태를 만들고 지켜야 했습니다.

---

## 2. 판단 기준

### 2-1. 적립을 승인 도메인에 종속시키려던 시도들

- **승인 트랜잭션 안에서 적립까지 함께 처리하는 방식**: 가장 단순해 보였지만, 승인 결정(moderation)과 보상 처리(reward)가 같은 트랜잭션에 묶이면 어느 한쪽의 실패가 다른 쪽에 직접 영향을 줍니다. 승인 도메인이 적립 도메인의 운영 상태에 의존하게 되는 구조였습니다.
- **잔액만 관리하는 방식**: "승인되면 포인트를 더한다" 정도로도 충분해 보였지만, 같은 승인 건 재처리 시 중복 방어, 누락과 차단의 구분, 취소 이력 추적 같은 질문에 답하기 어려웠습니다. 잔액만으로는 적립 상태를 설명할 수 없었습니다.
- **재계산으로 복원하는 방식**: 평점 집계 계층은 원본 기준이 살아 있으면 다시 계산해 맞출 수 있습니다. 그러나 적립은 승인된 기여가 보상 상태로 번역된 결과라서, 누락과 중복의 의미가 훨씬 직접적입니다. 적립이 빠지면 사용자는 자기 기여가 인정되지 않았다고 받아들이고, 중복되면 운영 정정 이슈가 아니라 보상 사고로 읽힙니다.

### 2-2. 판단 기준 — 도메인 경계와 정합성 수준

승인 결정의 Source of Truth는 moderation이고, 적립 원장의 정합성은 reward가 독립적으로 책임집니다. 두 도메인은 Outbox + RabbitMQ로 연결하되, 각자 자기 경계 안에서 정합성을 보장하는 구조로 나눴습니다.

정합성 수준은 의도적으로 조정했습니다. 2PC나 분산 트랜잭션처럼 두 도메인을 강하게 묶지는 않았습니다. 그 대신 moderation이 Outbox로 이벤트 유실을 막고, reward가 멱등성과 불변식으로 중복·누락을 방어하는 구조를 택했습니다. 평점 집계처럼 "결국 수렴하면 된다"는 접근은 맞지 않았지만, 금융 정산처럼 두 도메인이 동기적으로 묶여야 할 이유도 없었습니다. 승인 이벤트가 reward에 도달하면, 그 이후의 정합성은 reward가 자기 불변식으로 완전히 책임지는 구조가 이 서비스에 맞았습니다.

### 2-3. 최종 구조 — 불변식

| 불변식 | 의미 |
| --- | --- |
| 승인 1건 → 최대 1번 적립 | 같은 승인 이벤트가 재전달돼도 기존 결과로 수렴 |
| 서로 다른 승인 기여는 각각 적립 가능 | 보상 기준을 사용자 단위가 아니라 기여 단위로 |
| 한도 초과는 정책 차단 항목으로 남음 | 조용한 누락이 아니라 `CAP_EXCEEDED`로 기록 |
| 잘못된 적립은 덮어쓰지 않고 상쇄 | `REVERSAL` 항목으로 시간순 추적 가능 |

이 기준을 먼저 세우고 나니, 적립은 포인트 기능이 아니라 승인 단위 보상 상태를 설명하는 원장 경계로 읽히기 시작했습니다.

---

## 3. 구현 과정

### 3-1. 도메인 경계를 Outbox로 연결했습니다

moderation은 승인 결정과 함께 같은 트랜잭션 안에서 reward outbox에 이벤트를 기록합니다. 승인 사실이 outbox에 확정되는 순간까지가 moderation의 책임 범위입니다. reward는 RabbitMQ를 통해 이 이벤트를 비동기로 소비하고, 자기 도메인 안에서 적립 원장에 반영합니다.

이 구조에서 두 도메인은 이벤트로만 연결됩니다. moderation은 reward의 적립 결과를 알지 못하고, reward는 승인 결정 과정에 관여하지 않습니다. Outbox가 이벤트 유실을 막고, reward 내부 멱등성이 중복 소비를 흡수합니다.

### 3-2. 적립 입력을 검수 승인으로 고정했습니다

reward가 소비하는 이벤트의 기준은 `APPROVED` 상태의 검수 결과입니다. 승인 식별자 `approvalId`를 적립 입력 기준으로 삼아, 같은 승인 건을 같은 기여로 해석하도록 맞췄습니다. 이 기준이 있어야 적립·조회·취소가 모두 같은 대상을 가리킬 수 있습니다.

### 3-3. 중복 방어를 서비스와 저장소에 나눠 뒀습니다

Outbox + RabbitMQ 구조에서 같은 이벤트가 재전달될 수 있다는 전제 위에서, 서비스 계층에서는 기존 적립 결과가 있으면 그 결과를 그대로 반환하게 했습니다. 저장소에는 `(approvalId, campaignCode, entryType)` 고유 제약을 둬서 경쟁 상황에서도 중복 생성을 막았습니다. 사용자 경험 보호와 최종 데이터 보호선을 분리한 것입니다.

### 3-4. 한도 초과와 취소도 원장 항목으로 남겼습니다

적립이 일어나지 않았다고 해서 모두 같은 상태는 아닙니다. 실제 누락과 정책 차단은 운영 의미가 다릅니다. 그래서 한도 초과는 `CAP_EXCEEDED` 상태의 0포인트 항목으로 남기고, 잘못 적립된 건은 `REVERSAL` 항목으로 따로 기록했습니다. "왜 적립이 없었는가", "무엇이 나중에 되돌려졌는가"를 기존 값 수정 없이 시간순으로 설명할 수 있게 됐습니다.

---

## 4. 결과

이번 작업에서 가장 먼저 확인한 건 적립이 정말로 설명 가능한 상태가 됐는지였습니다. 동시에 도메인 경계가 의도대로 나뉘었는지도 함께 봤습니다.

| 확인 항목 | 기대 동작 |
| --- | --- |
| 같은 승인 이벤트 재전달 | 기존 결과로 수렴, 중복 생성 없음 |
| 같은 사용자의 서로 다른 승인 기여 | 각각 독립적으로 적립 |
| 한도 초과 | 조용한 실패 없이 `CAP_EXCEEDED` 항목으로 기록 |
| 잘못된 적립 취소 | 기존 값 수정 없이 `REVERSAL` 항목으로 상쇄 |
| moderation 장애 시 reward 영향 | 없음. 이미 outbox에 기록된 이벤트는 reward가 독립적으로 소비 |

이 작업으로 가장 또렷해진 건, 초기 카탈로그에서는 기여를 더 받는 것만큼 어떤 기여를 신뢰할지 분명히 보여 주는 것도 중요하다는 점이었습니다. 적립을 검수 기준으로 묶으면서, 기여 유도와 품질 통제를 같은 흐름 안에서 설명할 수 있게 됐습니다.

---

## 5. 남겨둔 것

이번 단계는 적립 원장 본체를 닫는 데 집중했고, 범위도 의도적으로 보수적으로 잡았습니다.

- **적립 입력 범위가 제한적입니다.** 현재 수동 `APPROVED`만 받습니다. `AUTO_APPROVED`는 다음 확장 범위입니다.
- **캠페인이 하나로 제한됩니다.** 기본 캠페인 `catalog_bootstrap_v1` 하나만 운영합니다. 복수 캠페인과 차등 적립 규칙은 별도로 설계해야 합니다.
- **Outbox 재전달 보장은 RabbitMQ 운영 장치에 기댑니다.** reward 내부 멱등성이 중복을 흡수하지만, outbox 메시지가 영구 실패할 경우의 복구 경로는 별도로 정의해야 합니다.
- **후속 단계가 열려 있습니다.** 적립 사용이나 외부 정산은 별도 경계로 남겨 뒀습니다.

여기서 닫은 것은 "보상을 붙였다"가 아니라, 도메인 경계를 나누고 그 경계 안에서 적립·차단·취소를 불변식으로 보호할 수 있는 적립 원장 본체였습니다.
