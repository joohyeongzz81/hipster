# 승인과 적립을 분리해 보상 상태를 설명하는 적립 원장

초기 카탈로그에서는 사용자 기여를 더 끌어올 장치가 필요해, 승인된 기여에 보상을 주는 방식을 떠올렸습니다. 그런데 이를 단순 포인트 가산으로 열어 두면, 무엇이 인정된 기여인지, 왜 적립이 막혔는지, 같은 승인 건이 왜 두 번 적립되지 않는지를 끝까지 설명하기 어려웠습니다.

승인 여부는 검수 도메인의 책임이었고, 적립 여부와 취소 이력은 적립 원장이 따로 설명해야 하는 상태였습니다. 그래서 검수는 승인 사실만 확정하고, 적립은 그 결과를 받아 보상 상태를 자기 원장에서 설명하는 구조가 더 맞았습니다. 이 문서는 왜 둘을 같은 트랜잭션에 묶지 않았는지, 그리고 분리한 뒤에도 적립 원장에서 어떤 불변식을 지켰는지 정리합니다.

---

## 1. 문제 상황

이 서비스는 사용자의 제출과 검수를 통해 카탈로그를 늘려 가는 구조라, 초기에 기여 유도 장치가 필요했습니다. 적립은 그 지점을 버티기 위한 장치였지만, 핵심은 "보상을 준다"가 아니라 "무엇을 보상으로 인정할 것인가"였습니다.

예를 들어 운영자가 어떤 제출을 승인했더라도, 그 기여가 이미 캠페인 한도를 넘었다면 적립은 막아야 할 수 있습니다. 이때 승인과 적립이 같은 트랜잭션에 묶여 있으면, 승인도 함께 실패시킬지 적립만 차단할지를 같은 흐름에서 결정해야 합니다. 하지만 비즈니스 의미는 다릅니다. 제출은 품질 기준상 승인될 수 있고, 보상은 정책 기준상 차단될 수 있습니다.

그래서 적립 입력은 제출이 아니라 검수 승인 결과여야 했습니다. 그리고 여기서부터는 평점 집계 계층과 다른 문제가 시작됐습니다. 적립은 원본 기준이 살아 있으면 다시 계산해 맞출 수 있는 파생 값이 아니었습니다. 승인 사실을 받아 별도 원장이 적립·차단·취소를 기록해야 했고, 그 상태는 적립 도메인 안에서 독립적으로 보호돼야 했습니다.

---

## 2. 판단 기준

승인은 이 기여를 인정할 것인가에 답하고, 적립은 인정된 기여를 어떤 보상 상태로 남길 것인가에 답합니다. 이 둘을 같은 트랜잭션으로 묶으면 승인 완료가 적립 정책과 저장 상태에 끌려다니고, 승인 상태만으로는 정책 차단과 취소 상쇄를 설명하기 어렵습니다. 그래서 검수 도메인은 승인 사실만 확정해 넘기고, 적립 도메인은 그 사실을 받아 적립·차단·취소를 자기 원장에서 설명하도록 책임을 나눴습니다.

경계를 나누면 검수는 품질 판단에 집중하고, 적립은 보상 상태를 자기 기준으로 독립적으로 운영할 수 있습니다. 다만 경계를 나눴다고 해서 적립 정합성까지 느슨하게 둘 수는 없었습니다. 같은 승인 건이 두 번 적립되거나 빠지는 순간, 사용자는 곧바로 보상 문제로 받아들이기 때문입니다. 그래서 도메인 사이에서는 승인 사실 유실을 막고, 적립 도메인 안에서는 멱등성과 불변식으로 상태를 강하게 보호했습니다.

### 적립 원장에서 지키기로 한 불변식

| 불변식 | 의미 |
| --- | --- |
| 승인 1건 → 최대 1번 적립 | 같은 승인 이벤트가 재전달돼도 기존 결과로 수렴 |
| 서로 다른 승인 기여는 각각 적립 가능 | 보상 기준을 사용자 단위가 아니라 기여 단위로 |
| 한도 초과는 정책 차단 항목으로 남음 | 조용한 누락이 아니라 `CAP_EXCEEDED`로 기록 |
| 잘못된 적립은 덮어쓰지 않고 상쇄 | `REVERSAL` 항목으로 시간순 추적 가능 |

적립은 포인트 기능이 아니라 승인 단위 보상 상태를 설명하는 원장 경계가 됐습니다.

---

## 3. 구현 과정

### 3-1. 도메인 경계를 비동기로 연결했습니다

검수와 적립을 같은 트랜잭션으로 묶지 않기로 한 이상, 두 도메인 사이에서는 승인 사실이 유실되지 않게 넘기는 장치가 먼저 필요했습니다. 검수 도메인은 승인 결정과 함께 적립 전달 기록을 같은 트랜잭션 안에 남기고, 적립 도메인은 메시지 브로커를 통해 이 이벤트를 비동기로 소비해 자기 원장에 반영합니다. 구현은 `Outbox`와 `RabbitMQ`를 사용했습니다.

이 구조에서 두 도메인은 이벤트로만 연결됩니다. 검수 도메인은 적립 결과를 알지 못하고, 적립 도메인은 승인 결정 과정에 관여하지 않습니다. `Outbox`가 승인 사실 유실을 막고, 적립 도메인 내부 멱등성이 중복 소비를 흡수합니다.

### 3-2. 적립 입력을 검수 승인으로 고정했습니다

적립 도메인이 소비하는 이벤트의 기준은 `APPROVED` 상태의 검수 결과입니다. 승인 식별자 `approvalId`를 적립 입력 기준으로 삼아, 같은 승인 건을 같은 기여로 해석하도록 맞췄습니다. 이 기준이 있어야 적립·조회·취소가 모두 같은 대상을 가리킬 수 있습니다.

### 3-3. 중복 방어를 서비스와 저장소에 나눠 뒀습니다

같은 승인 이벤트가 재전달될 수 있다는 전제는 놓을 수 없습니다. 서비스 계층에서는 기존 적립 결과가 있으면 그 결과를 그대로 반환하게 했고, 저장소에는 `(approvalId, campaignCode, entryType)` 고유 제약을 둬서 경쟁 상황에서도 중복 생성을 막았습니다. 사용자 경험 보호와 최종 데이터 보호선을 분리한 것입니다.

### 3-4. 한도 초과와 취소도 원장 항목으로 남겼습니다

적립이 일어나지 않았다고 해서 모두 같은 상태는 아닙니다. 실제 누락과 정책 차단은 운영 의미가 다릅니다. 그래서 한도 초과는 `CAP_EXCEEDED` 상태의 0포인트 항목으로 남기고, 잘못 적립된 건은 `REVERSAL` 항목으로 따로 기록했습니다. "왜 적립이 없었는가", "무엇이 나중에 되돌려졌는가"를 기존 값 수정 없이 시간순으로 설명할 수 있게 됐습니다.

---

## 4. 결과

승인과 적립을 분리한 뒤에도, 적립 원장이 자기 경계 안에서 보상 상태를 설명하고 보호하는지는 아래 시나리오로 확인했습니다.

| 검증 시나리오 | 확인 결과 |
| --- | --- |
| 승인됐지만 캠페인 한도를 넘은 기여 | 승인 상태는 유지됐고, 적립 원장에는 `CAP_EXCEEDED` 항목이 남았습니다. |
| 같은 승인 이벤트 재전달 | 기존 적립 결과로 수렴했고, 새 적립 항목은 생성되지 않았습니다. |
| 같은 사용자의 서로 다른 승인 기여 | 승인 단위로 각각 별도 적립이 생성됐습니다. |
| 잘못된 적립 취소 | 기존 항목을 수정하지 않고 `REVERSAL` 항목으로 상쇄했습니다. |

이 작업으로 적립은 단순 포인트 가산이 아니라, 어떤 승인 기여가 왜 적립됐고 왜 차단됐고 왜 되돌려졌는지를 설명할 수 있는 원장 경계가 됐습니다. 검수와 적립을 분리하되, 적립 경계 안에서는 상태를 느슨하게 두지 않고 불변식으로 보호하는 구조가 이 문서의 핵심이었습니다.
